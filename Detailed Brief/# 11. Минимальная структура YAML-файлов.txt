# 11. Минимальная структура YAML-файлов

## 11.1. Назначение этой главы

Данная глава описывает минимальную структуру YAML-файлов, используемых в первой версии продукта. Её задача — зафиксировать предварительные форматы основных артефактов системы так, чтобы команда разработки могла опираться на них при проектировании загрузки, сохранения, адресации и пользовательского интерфейса.

В первой версии в YAML должны существовать как минимум три основных типа файлов:

* **manifest проекта**;
* **модель**;
* **нотация**.

Эти форматы не являются окончательной спецификацией всей будущей платформы. Они описывают только минимально необходимый слой, достаточный для MVP.

---

## 11.2. Общие принципы YAML-форматов MVP

Перед описанием конкретных файлов важно зафиксировать несколько общих принципов.

### 11.2.1. YAML должен оставаться читаемым человеком

Файлы должны быть пригодны для ручного просмотра и редактирования во внешнем редакторе, например в Visual Studio Code.

### 11.2.2. Форматы должны быть минимальными

В MVP не нужно включать в YAML всё потенциально возможное. Достаточно зафиксировать только те поля, которые реально поддерживают сценарии первой версии.

### 11.2.3. Идентичность не должна зависеть от display name

Во всех структурах должны быть поля, отвечающие за устойчивую идентичность сущности, отдельно от её отображаемого имени.

### 11.2.4. Freeform и typed должны использовать одну общую форму модели

Не нужно вводить два разных формата файла модели. Достаточно одного формата, в котором режим модели задаётся полем.

### 11.2.5. Ссылки должны быть явными

Связи между моделями, нодами и frame должны описываться явно через ссылки, а не восстанавливаться косвенно из положения элементов на диаграмме.

---

## 11.3. YAML-файл manifest проекта

## 11.3.1. Назначение файла

Manifest описывает проект как отдельную рабочую область. Это входной файл, по которому система понимает, что перед ней именно проект, а не просто папка с произвольными YAML-документами.

## 11.3.2. Минимальные поля manifest

В первой версии manifest может содержать:

* `kind` — тип документа;
* `version` — версия формата;
* `name` — имя проекта;
* `defaultModel` — стартовая модель проекта, если задана;
* `meta` — необязательный блок расширения.

## 11.3.3. Рекомендуемый пример

```yaml
kind: project
version: 1
name: My Research Project
defaultModel: models/main.yaml

meta:
  description: Experimental workspace for framework exploration
```

## 11.3.4. Комментарии к структуре

* `kind: project` нужен, чтобы система различала типы YAML-документов.
* `version` нужен как задел на эволюцию формата.
* `defaultModel` в MVP опционален, но полезен как точка входа.
* `meta` не обязателен, но даёт место для постепенного расширения без ломки структуры.

---

## 11.4. YAML-файл модели

## 11.4.1. Назначение файла

Файл модели описывает отдельную диаграмму. Это основной рабочий артефакт системы, который открывается на canvas и редактируется пользователем.

Один и тот же формат должен поддерживать:

* freeform-модели;
* typed-модели;
* модели, созданные через drill-down;
* модели, созданные через step-up.

## 11.4.2. Минимальные верхнеуровневые поля модели

В первой версии модель должна содержать как минимум:

* `kind` — тип документа;
* `version` — версия формата;
* `id` — внутренний идентификатор модели;
* `name` — отображаемое имя модели;
* `mode` — `freeform` или `typed`;
* `notation` — ссылка на нотацию, если модель типизирована;
* `nodes` — список объектов;
* `edges` — список связей;
* `frames` — список frame;
* `meta` — необязательные дополнительные сведения.

## 11.4.3. Рекомендуемый пример верхнего уровня

```yaml
kind: model
version: 1
id: model_main
name: Main Map
mode: freeform
notation: null

nodes: []
edges: []
frames: []

meta: {}
```

---

## 11.5. Структура node внутри модели

## 11.5.1. Назначение node

Node — базовая смысловая единица диаграммы. Она должна хранить как визуально необходимые данные, так и минимальный семантический контекст.

## 11.5.2. Минимальные поля node

Для MVP у node должны быть как минимум:

* `id` — стабильный ID внутри модели;
* `label` — отображаемое имя;
* `description` — необязательное текстовое описание;
* `type` — тип объекта, если назначен;
* `position` — координаты на canvas;
* `drilldowns` — список ссылок на детализирующие модели;
* `style` — минимальные визуальные данные, если нужны;
* `meta` — дополнительный блок расширения.

## 11.5.3. Рекомендуемый пример node

```yaml
- id: n1
  label: User Need
  description: Core user need identified during exploration
  type: null

  position:
    x: 120
    y: 80

  drilldowns: []

  style:
    color: "#F5E6A8"

  meta: {}
```

## 11.5.4. Комментарии к структуре

* `id` обязателен и не должен зависеть от `label`.
* `type` может быть `null` в freeform-модели.
* `drilldowns` должен быть списком даже тогда, когда детализация одна.
* `style` можно оставить минимальным, поскольку визуальный язык MVP простой.

---

## 11.6. Структура edge внутри модели

## 11.6.1. Назначение edge

Edge описывает направленную связь между двумя node внутри одной модели.

## 11.6.2. Минимальные поля edge

Для MVP достаточно следующих полей:

* `id` — идентификатор связи;
* `source` — ID исходной node;
* `target` — ID целевой node;
* `label` — необязательная подпись связи;
* `meta` — дополнительный блок.

## 11.6.3. Рекомендуемый пример edge

```yaml
- id: e1
  source: n1
  target: n2
  label: relates to
  meta: {}
```

## 11.6.4. Комментарии к структуре

* Тип связи в MVP не обязателен.
* Даже если `label` пустой, связь должна оставаться самостоятельной сущностью.
* Связь существует только внутри одной модели.

---

## 11.7. Структура frame внутри модели

## 11.7.1. Назначение frame

Frame объединяет группу node и поддерживает сценарий step-up.

## 11.7.2. Минимальные поля frame

В первой версии frame должен содержать:

* `id` — стабильный ID;
* `label` — имя frame;
* `description` — описание;
* `nodes` — список ID входящих node;
* `stepUp` — ссылка на верхнеуровневую модель, если она уже создана;
* `bounds` — координаты и размеры рамки на canvas;
* `meta` — дополнительные сведения.

## 11.7.3. Рекомендуемый пример frame

```yaml
- id: f1
  label: Research Cluster
  description: Group of concepts that form one upper-level object

  nodes:
    - n1
    - n2
    - n3

  bounds:
    x: 80
    y: 40
    width: 420
    height: 260

  stepUp:
    model: models/upper-level/system-context.yaml
    nodeId: n1

  meta: {}
```

## 11.7.4. Комментарии к структуре

* `nodes` описывает состав frame явно.
* `bounds` нужны для восстановления визуального состояния.
* `stepUp` опционален: пока step-up не выполнен, поле может быть `null`.

---

## 11.8. Структура ссылок на модели

## 11.8.1. Зачем нужен единый формат ссылки

Поскольку в системе есть drill-down и step-up, полезно заранее унифицировать способ ссылки на модель или на сущность внутри модели.

## 11.8.2. Минимальная структура model reference

Для MVP достаточно следующей формы:

* `model` — относительный путь к файлу модели;
* `nodeId` — если ссылка идёт на конкретную node;
* `frameId` — если ссылка идёт на конкретный frame.

Не все поля используются одновременно.

## 11.8.3. Пример ссылки на drill-down

```yaml
model: models/details/user-need.yaml
```

## 11.8.4. Пример ссылки на node в другой модели

```yaml
model: models/upper-level/system-context.yaml
nodeId: n1
```

## 11.8.5. Пример ссылки на frame в другой модели

```yaml
model: models/main.yaml
frameId: f1
```

---

## 11.9. Структура drill-down внутри node

## 11.9.1. Назначение

Так как одна node может иметь несколько детализаций, drill-down должен описываться списком ссылок.

## 11.9.2. Минимальная форма элемента drill-down

Каждый элемент списка drill-down может содержать:

* `model` — путь к модели детализации;
* `label` — необязательное пользовательское имя этой детализации;
* `notationHint` — необязательное указание рекомендуемой нотации;
* `meta` — дополнительные сведения.

## 11.9.3. Пример

```yaml
drilldowns:
  - model: models/details/user-need.yaml
    label: Main Detail
    notationHint: freeform
    meta: {}

  - model: models/details/user-need-workflow.yaml
    label: Workflow View
    notationHint: notations/workflow-map-notation.yaml
    meta: {}
```

## 11.9.4. Комментарии

* `notationHint` не обязателен, но может быть полезен в UI.
* Один объект может иметь несколько представлений — это должно сразу быть видно в структуре.
* Даже если детализация одна, она всё равно хранится как элемент списка.

---

## 11.10. Полный пример YAML-модели

Ниже — пример минимальной, но уже содержательной модели.

```yaml
kind: model
version: 1
id: model_main
name: Main Map
mode: freeform
notation: null

nodes:
  - id: n1
    label: User Need
    description: Core user need
    type: need
    position:
      x: 120
      y: 80
    drilldowns:
      - model: models/details/user-need.yaml
        label: Need Details
        notationHint: freeform
        meta: {}
    style:
      color: "#F5E6A8"
    meta: {}

  - id: n2
    label: Solution Concept
    description: Draft solution
    type: concept
    position:
      x: 420
      y: 100
    drilldowns: []
    style:
      color: "#B8D8F8"
    meta: {}

edges:
  - id: e1
    source: n1
    target: n2
    label: leads to
    meta: {}

frames:
  - id: f1
    label: Opportunity Space
    description: Main cluster
    nodes:
      - n1
      - n2
    bounds:
      x: 80
      y: 40
      width: 500
      height: 220
    stepUp:
      model: models/upper-level/opportunity-space.yaml
      nodeId: n1
    meta: {}

meta: {}
```

## 11.11. YAML-файл нотации

## 11.11.1. Назначение файла

Файл нотации описывает минимальный язык типов объектов, на основе которого можно создавать typed-модели.

В MVP нотация должна быть лёгкой и не перегруженной. Она не обязана описывать всё поведение системы. Её основная задача — зафиксировать набор типов объектов и их базовые характеристики.

## 11.11.2. Минимальные верхнеуровневые поля нотации

В первой версии notation должна содержать как минимум:

* `kind` — тип документа;
* `version` — версия формата;
* `id` — идентификатор нотации;
* `name` — отображаемое имя;
* `objectTypes` — список типов объектов;
* `meta` — дополнительные сведения.

## 11.11.3. Рекомендуемый пример верхнего уровня

```yaml
kind: notation
version: 1
id: workflow_map
name: Workflow Map

objectTypes: []

meta: {}
```

---

## 11.12. Структура object type внутри нотации

## 11.12.1. Назначение object type

Object type описывает тип объектов, который пользователь может использовать на typed-диаграмме.

## 11.12.2. Минимальные поля object type

Для MVP у object type должны быть:

* `id` — внутренний идентификатор типа;
* `label` — отображаемое имя;
* `description` — необязательное описание;
* `style` — минимальные визуальные свойства;
* `drilldownSuggestions` — необязательный список рекомендуемых нотаций для детализации;
* `meta` — дополнительные сведения.

## 11.12.3. Пример object type

```yaml
- id: need
  label: User Need
  description: A need identified in the modeled space

  style:
    color: "#F5E6A8"

  drilldownSuggestions:
    - freeform
    - notations/workflow-map-notation.yaml

  meta: {}
```

## 11.12.4. Комментарии к структуре

* `id` должен быть коротким и устойчивым.
* `label` — это пользовательское имя типа.
* `style.color` достаточно для MVP.
* `drilldownSuggestions` не обязателен, но помогает UI предлагать варианты детализации.

---

## 11.13. Полный пример YAML-нотации

```yaml
kind: notation
version: 1
id: workflow_map
name: Workflow Map

objectTypes:
  - id: need
    label: User Need
    description: A need identified in the modeled space
    style:
      color: "#F5E6A8"
    drilldownSuggestions:
      - freeform
    meta: {}

  - id: concept
    label: Solution Concept
    description: A conceptual solution or idea
    style:
      color: "#B8D8F8"
    drilldownSuggestions:
      - freeform
    meta: {}

  - id: actor
    label: Actor
    description: A participant or role in the structure
    style:
      color: "#D9C2F0"
    drilldownSuggestions:
      - freeform
    meta: {}

meta: {}
```

---

## 11.14. Как freeform и typed отражаются в YAML

## 11.14.1. Freeform-модель

Freeform-модель определяется так:

* `mode: freeform`
* `notation: null`

Пример:

```yaml
mode: freeform
notation: null
```

## 11.14.2. Typed-модель

Typed-модель определяется так:

* `mode: typed`
* `notation` содержит ссылку на YAML-нотацию

Пример:

```yaml
mode: typed
notation: notations/workflow-map-notation.yaml
```

## 11.14.3. Смысл этого решения

Это позволяет использовать одну и ту же структуру модели в обоих случаях и не раздваивать архитектуру MVP.

---

## 11.15. Минимальные обязательные поля

Чтобы MVP был устойчивым, полезно заранее зафиксировать, какие поля считаются обязательными.

### 11.15.1. Для manifest

Обязательны:

* `kind`
* `version`
* `name`

### 11.15.2. Для модели

Обязательны:

* `kind`
* `version`
* `id`
* `name`
* `mode`
* `nodes`
* `edges`
* `frames`

### 11.15.3. Для node

Обязательны:

* `id`
* `label`
* `position`

### 11.15.4. Для edge

Обязательны:

* `id`
* `source`
* `target`

### 11.15.5. Для frame

Обязательны:

* `id`
* `label`
* `nodes`

### 11.15.6. Для notation

Обязательны:

* `kind`
* `version`
* `id`
* `name`
* `objectTypes`

### 11.15.7. Для object type

Обязательны:

* `id`
* `label`

---

## 11.16. Что сознательно не описывается в YAML MVP

Чтобы не перегрузить первую версию, в YAML пока не нужно включать:

* типизированные связи;
* формальные ограничения допустимости связей;
* сложную геометрию layout;
* multi-user метаданные;
* историю изменений;
* механизмы синхронизации;
* execution semantics;
* сложные правила наследования нотаций.

Эти элементы могут появиться позже, но в MVP они только ухудшат читаемость и усложнят реализацию.

---

## 11.17. Допустимые расширения без ломки формата

Уже сейчас структура YAML может быть построена так, чтобы позже естественно расширяться.

Например, в будущем можно будет добавить:

* новые поля в `meta`;
* типизацию `edges`;
* дополнительные свойства `objectTypes`;
* визуальные настройки shape;
* ограничения нотации;
* ссылки на внешние ресурсы;
* библиотеки reusable types.

Именно поэтому полезно уже в MVP оставить у основных сущностей поле `meta`.

---

## 11.18. Итоговый смысл этой главы

Минимальная структура YAML-файлов первой версии должна отражать ту же основную идею, что и сам продукт: **простота, прозрачность и постепенная формализация**.

Manifest задаёт проект как рабочее пространство.
Model задаёт диаграмму как основной носитель структуры.
Notation задаёт возникающий язык типов.
Node, Edge и Frame фиксируют минимальные элементы визуального и смыслового слоя.
Drill-down и step-up выражаются через явные ссылки между моделями.

Таким образом, YAML в MVP — это не просто технический формат хранения, а текстовое выражение того же самого пространства уровней, переходов и возникающих нотаций, с которым пользователь работает в интерфейсе.
