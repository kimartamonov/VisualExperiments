# 14. Технические ориентиры

## 14.1. Назначение этой главы

Данная глава фиксирует базовые технические ориентиры для разработки первой версии продукта. Её задача — не задать окончательную архитектуру всей будущей платформы, а определить минимальный и разумный технологический контур MVP, который соответствует уже зафиксированной концепции продукта.

Технические ориентиры в данном случае нужны для того, чтобы команда:

* не проектировала избыточно сложную систему раньше времени;
* выбирала решения, совместимые с исследовательским характером продукта;
* сразу закладывала прозрачную структуру проекта, моделей и файлов;
* опиралась на достаточно простую, но расширяемую технологическую базу.

Главный принцип этой главы:
**первая версия должна быть технологически простой, но не случайной**.

---

## 14.2. Общий технический подход

Первая версия продукта должна разрабатываться как веб-приложение с серверным запуском и браузерным интерфейсом. Это соответствует основной цели MVP: быстро получить рабочую визуальную песочницу, доступную по адресу, с возможностью открывать проекты, редактировать модели и сохранять их как файловую структуру.

Технически система в первой версии должна строиться вокруг четырёх базовых слоёв:

1. **Frontend** — визуальный редактор, дерево проекта, панель свойств, навигация между моделями.
2. **Canvas layer** — работа с диаграммами, node, edge, frame и переходами между уровнями.
3. **Persistence layer** — чтение и запись YAML-файлов проекта.
4. **Project/file access layer** — доступ к структуре проекта как к набору папок и файлов.

Такое разделение соответствует уже сложившейся логике продукта:
UI работает не с абстрактной внутренней базой, а с реальным пространством проектных файлов.

---

## 14.3. Технологический стек первой версии

## 14.3.1. Frontend

Для frontend первой версии рекомендуется использовать:

* **React**
* **TypeScript**

### Почему это соответствует продукту

* React хорошо подходит для построения сложного интерактивного интерфейса с несколькими зонами состояния.
* TypeScript даёт удобную типизацию для сущностей вроде Project, Model, Node, Frame, Notation и ссылок между ними.
* Такой стек естественно сочетается с React Flow, который уже является выбранным референсом для центрального canvas.

### Практический смысл

Это позволяет быстро собрать MVP, не усложняя архитектуру и не отрываясь от уже намеченного визуального подхода.

---

## 14.3.2. Библиотека диаграмм

Для центрального визуального слоя рекомендуется использовать:

* **React Flow**

### Почему это соответствует продукту

React Flow подходит, потому что он уже близок к тому способу мышления, который нужен в MVP:

* есть node и edge;
* можно строить интерактивный canvas;
* удобно поддерживать drag-and-drop, выделение, перемещение;
* можно постепенно обрастать кастомной логикой;
* он ближе к нужному “Sirius Web / FigJam-like” направлению, чем тяжёлые enterprise diagram engines.

### Ограничение

React Flow сам по себе не решает семантику продукта.
Он должен рассматриваться как графический substrate, а не как готовая модельная платформа.

---

## 14.3.3. Backend

Для первой версии backend может быть минимальным и выполнять прежде всего задачи:

* открытия списка проектов;
* чтения файлов проекта;
* записи YAML-файлов;
* создания новых файлов моделей и нотаций;
* обслуживания базовой навигации по структуре проекта.

Для MVP допустим простой backend на:

* **Node.js**
* **TypeScript**

### Почему это соответствует продукту

* единый язык между frontend и backend упрощает разработку;
* файловые операции и работа с YAML удобно реализуются в Node.js;
* первая версия не требует сложной серверной логики, очередей, оркестрации и тяжёлых интеграций.

---

## 14.3.4. Формат хранения

В качестве формата хранения в первой версии должен использоваться:

* **YAML**

Для:

* manifest проекта;
* моделей;
* нотаций.

### Почему это соответствует продукту

* формат читаем человеком;
* легко редактируется вручную;
* хорошо подходит для исследовательского и прозрачного workflow;
* поддерживает постепенное усложнение структуры без отказа от понятности.

---

## 14.4. Архитектурные приоритеты MVP

## 14.4.1. Приоритет 1. Простота важнее полноты

Архитектура первой версии должна быть достаточно простой, чтобы команда могла быстро собрать сквозной сценарий MVP. Не нужно в первой версии строить “платформу на вырост” с большим количеством абстракций, если эти абстракции не поддерживают непосредственно демонстрационный сценарий.

### Практический вывод

Лучше:

* простой backend;
* явные YAML-структуры;
* прямые model references;
* ограниченный набор UI-компонентов.

Чем:

* сложная внутренняя DSL-архитектура;
* избыточный event sourcing;
* продвинутая схема синхронизации;
* раннее усложнение meta-layer.

---

## 14.4.2. Приоритет 2. Файлы — часть архитектуры, а не побочный экспорт

Система не должна скрывать модели во внутреннем формате, который потом “при желании” экспортируется.
Наоборот, YAML-файлы проекта должны быть первичным источником устойчивого состояния.

### Практический вывод

Нужно проектировать так, чтобы:

* открытие проекта читало файловую структуру;
* изменения в UI отражались в YAML;
* ссылки между моделями были прозрачными;
* проект можно было понимать и без запуска приложения.

---

## 14.4.3. Приоритет 3. Один мир для freeform и typed

Архитектура не должна раскалывать систему на две почти независимые подсистемы:

* “редактор свободных досок”;
* “редактор типизированных моделей”.

Для MVP это один и тот же редактор, одна и та же сущность `Model`, один и тот же canvas, одна и та же логика переходов.

### Практический вывод

Различие между freeform и typed должно жить:

* в данных модели;
* в доступных действиях UI;
* в наличии или отсутствии notation reference.

Но не в двух разных архитектурных мирах.

---

## 14.4.4. Приоритет 4. Переходы между уровнями — часть ядра

Drill-down и step-up не должны быть реализованы как случайные ссылки “где-то в боковой панели”.
Они являются частью центральной логики продукта.

### Практический вывод

Архитектурно нужно сразу предусмотреть:

* явные ссылки между моделями;
* устойчивую адресацию node и frame;
* восстановление контекста перехода;
* двустороннюю навигацию между уровнями.

Иначе продукт быстро потеряет свою главную концептуальную особенность.

---

## 14.5. Предварительная структура приложения

На уровне крупных блоков первая версия может быть организована следующим образом:

### 1. Project Browser Layer

Отвечает за:

* список проектов;
* открытие проекта;
* отображение дерева файлов.

### 2. Editor Workspace Layer

Отвечает за:

* layout трёхпанельного интерфейса;
* текущую открытую модель;
* навигацию между моделями.

### 3. Diagram Engine Layer

Отвечает за:

* работу canvas;
* node / edge / frame;
* выделение;
* drag-and-drop;
* визуальное состояние.

### 4. Model Semantics Layer

Отвечает за:

* drill-down;
* step-up;
* типизацию;
* переход freeform → typed;
* создание нотации из модели.

### 5. Persistence Layer

Отвечает за:

* чтение YAML;
* запись YAML;
* создание файлов;
* разрешение ссылок по путям.

Это не жёсткая модульная схема, а ориентир для того, чтобы команда не смешивала UI, файловую работу и смысловую логику в одном неразличимом слое.

---

## 14.6. Требования к frontend-архитектуре

## 14.6.1. Состояние интерфейса должно быть разделено по уровням

В первой версии полезно различать как минимум:

* состояние проекта;
* состояние текущей модели;
* состояние выбора на canvas;
* состояние навигации между уровнями;
* состояние панели свойств.

### Почему это важно

Иначе логика быстро начнёт смешиваться: выбор node, открытие модели, смена проекта и переход по drill-down начнут обрабатываться как одно и то же.

---

## 14.6.2. Canvas должен быть максимально “тонким”

Сама диаграммная библиотека не должна брать на себя бизнес-смысл продукта.

### Практический вывод

React Flow должен использоваться как:

* средство отображения;
* средство манипуляции node и edge;
* механизм пользовательского взаимодействия на полотне.

Но логика:

* что такое frame,
* что делает step-up,
* как хранится drill-down,
* как создаётся notation

должна находиться не внутри произвольных canvas-хаков, а в более явном слое модели приложения.

---

## 14.6.3. Панель свойств должна быть контекстной и управляемой данными

Правая панель должна строиться не как набор несвязанных форм, а как отражение текущего выделенного объекта.

### Практический вывод

Нужна логика вида:

* если выбрана node — показываем редактор node;
* если выбран frame — показываем редактор frame;
* если ничего не выбрано — показываем редактор модели;
* если выбрана edge — показываем свойства связи.

Это позволит расширять систему дальше, не ломая интерфейсную логику.

---

## 14.7. Требования к backend-архитектуре

## 14.7.1. Backend в MVP должен быть файлово-ориентированным

На данном этапе не нужно строить сложную базу данных моделей.

### Практический вывод

Backend должен уметь:

* перечислять проекты;
* читать manifest;
* читать YAML-модели;
* писать YAML-модели;
* создавать новые файлы;
* разрешать относительные ссылки между моделями.

---

## 14.7.2. Backend не должен навязывать тяжёлую серверную доменную модель

Поскольку первая версия — это исследовательская песочница, нет необходимости превращать backend в сложный orchestration core.

### Практический вывод

Server side в MVP — это прежде всего:

* project/file service;
* YAML persistence service;
* minimal API for editor operations.

А не:

* workflow runtime;
* distributed collaboration engine;
* rules execution system.

---

## 14.7.3. Серверный API должен быть ориентирован на сущности продукта

Даже если API будет минимальным, он должен мыслиться в терминах:

* project;
* model;
* notation;
* node;
* frame;
* links between models.

А не только в терминах “загрузить произвольный файл / сохранить произвольный файл”.

### Почему это важно

Потому что иначе смысловая логика продукта уйдёт в frontend и станет слишком хрупкой.

---

## 14.8. Работа с YAML

## 14.8.1. YAML должен быть устойчиво сериализуемым

Система должна читать и записывать YAML так, чтобы структура оставалась предсказуемой и понятной человеку.

### Практический вывод

Желательно:

* сохранять стабильный порядок ключей;
* не делать слишком агрессивных автоматических преобразований;
* не терять необязательные блоки `meta`;
* не разрушать структуру при незначительных изменениях.

---

## 14.8.2. Ручное редактирование YAML должно оставаться допустимым

Поскольку нотации в первой версии предполагается править в VS Code, система должна мирно сосуществовать с внешними изменениями файлов.

### Практический вывод

Полезно учитывать:

* повторное чтение файлов после внешнего изменения;
* аккуратную обработку ошибок YAML;
* понятные сообщения при невозможности загрузить документ.

---

## 14.8.3. Схема YAML должна быть мягко-валидируемой

Хотя в MVP не нужна тяжёлая формальная валидация моделей, минимальная проверка структуры файлов всё же полезна.

### Практический вывод

Система должна проверять:

* `kind`;
* `version`;
* обязательные поля;
* базовую корректность ссылок.

Но не должна превращать это в жёсткий бюрократический барьер для исследовательской работы.

---

## 14.9. Хранение и адресация

## 14.9.1. Основной ключ модели — путь к файлу

В первой версии ссылка на модель должна опираться на относительный путь внутри проекта.

### Почему это разумно

* просто;
* прозрачно;
* легко отлаживать;
* хорошо согласуется с файловой природой workspace.

---

## 14.9.2. Основной ключ node и frame — локальный ID внутри модели

Для node и frame достаточно устойчивых локальных идентификаторов.

### Практический вывод

Адресация вида:

* `model path + node id`
* `model path + frame id`

достаточна для drill-down и step-up в первой версии.

---

## 14.9.3. Нужно избегать раннего введения глобальных универсальных ID

Для MVP не нужно вводить тяжёлую систему глобальных идентификаторов на всю платформу.

### Почему это важно

Это увеличит сложность раньше времени и не даст ощутимой пользы для главного сценария первой версии.

---

## 14.10. Технические ограничения, которые полезно принять заранее

## 14.10.1. Нет real-time collaboration

Первая версия должна быть single-user oriented по своей технической модели.

## 14.10.2. Нет внутреннего version control

Нет необходимости сразу строить историю изменений, ветки и diff UI.

## 14.10.3. Нет базы данных как обязательной центральной зависимости

Файловое хранение достаточно для MVP.

## 14.10.4. Нет runtime-компиляции моделей в исполняемые процессы

Интеграция с Windmill в этой версии остаётся за пределами реализации.

## 14.10.5. Нет сложного plugin system

На старте не нужно строить универсальную расширяемую платформу с модульной загрузкой произвольных editor behaviors.

Эти ограничения важны не потому, что они “плохие”, а потому, что они помогают сохранить ясность первой версии.

---

## 14.11. UX-референсы и визуальные ориентиры

С точки зрения взаимодействия первая версия может ориентироваться на сочетание трёх интуиций:

### 1. Sirius Web

Как референс более model-oriented интерфейса.

### 2. React Flow

Как техническая и UX-основа canvas.

### 3. FigJam / Miro

Как референс лёгкости первого шага и свободного пространственного мышления.

### Практический вывод

Интерфейс MVP должен быть:

* проще, чем классические CASE-tools;
* структурнее, чем обычная белая доска;
* легче, чем тяжёлые modeling suites.

---

## 14.12. Требования к расширяемости

Хотя MVP должен быть простым, некоторые расширения стоит учитывать заранее на уровне направлений, а не полной реализации.

### В будущем может понадобиться:

* визуальный редактор нотаций;
* типизированные связи;
* richer shape system;
* логическое дерево моделей;
* shared instances;
* экспорт / трансформация в другие среды;
* интеграция с Windmill;
* шаблоны и библиотеки нотаций.

### Практический вывод

Поэтому архитектура MVP должна быть:

* достаточно простой для быстрой реализации;
* достаточно ясной, чтобы потом не пришлось всё переписывать из-за изначально спутанных сущностей.

---

## 14.13. Минимальный технический результат первой версии

С технической точки зрения MVP можно считать успешным, если система умеет:

* запускаться как веб-приложение;
* показывать список проектов;
* открывать проект как файловое workspace;
* читать и сохранять YAML-модели;
* отображать модель на React Flow canvas;
* поддерживать node, edge и frame;
* создавать и открывать drill-down модели;
* выполнять step-up;
* создавать notation YAML;
* создавать typed-модель на основе notation;
* восстанавливать проект после повторного открытия.

Это и есть минимальный технический контур, который соответствует продуктовой идее первой версии.

---

## 14.14. Итоговый смысл этой главы

Технические ориентиры первой версии должны удерживать одну простую мысль: MVP — это не прототип “на скорую руку” и не заготовка под всю платформу сразу. Это достаточно ясная и прозрачная система, в которой технологии подчинены главной идее продукта:

* свободное моделирование;
* переходы между уровнями;
* поздняя типизация;
* рождение нотации из практики;
* повторное моделирование по возникшему языку.

Поэтому техническая реализация должна быть лёгкой, файлово-прозрачной, React/TypeScript-ориентированной и достаточно чистой по слоям, чтобы не потерять эту идею под тяжестью преждевременной инженерной универсальности.
