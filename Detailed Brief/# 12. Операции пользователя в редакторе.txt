# 12. Операции пользователя в редакторе

## 12.1. Назначение этой главы

Данная глава описывает основные операции, которые пользователь может выполнять в редакторе первой версии продукта. Если предыдущие главы фиксировали смысл продукта, его сущности, сценарии и файловую структуру, то здесь нужно явно показать, **какие действия поддерживает интерфейс и над какими объектами эти действия совершаются**.

Для MVP это особенно важно, потому что продукт строится не вокруг большой библиотеки возможностей, а вокруг ограниченного, но связного набора операций, которые должны поддержать главный цикл работы:

**создать проект → создать модель → построить структуру → детализировать / укрупнить → назначить типы → создать нотацию → продолжить моделирование**

Иными словами, эта глава описывает не все потенциальные функции будущей платформы, а именно тот минимальный набор действий, который делает первую версию рабочей песочницей.

---

## 12.2. Общий принцип набора операций

В первой версии операции редактора должны соответствовать трём уровням работы пользователя:

### 1. Операции над проектом

Открытие, создание и навигация по проектному пространству.

### 2. Операции над моделью

Создание, открытие, сохранение и настройка отдельной диаграммы.

### 3. Операции над содержимым модели

Работа с node, edge, frame, drill-down, step-up и типизацией.

Такое деление важно, потому что в данном продукте пользователь работает не только “на canvas”, но и в более широком пространстве проектов и связанных моделей.

---

## 12.3. Операции уровня проекта

## 12.3.1. Создать проект

### Описание операции

Пользователь создаёт новый пустой проект через экран списка проектов.

### Результат операции

Система создаёт:

* папку проекта;
* `project.yaml`;
* каталог `models/`;
* при необходимости — базовую структуру для будущих нотаций.

### Значение для MVP

Это стартовая операция, без которой пользователь не может начать осмысленную работу в системе.

---

## 12.3.2. Открыть проект

### Описание операции

Пользователь выбирает существующий проект и входит в него.

### Результат операции

Открывается основной экран редактора, загружается дерево проекта и становится доступна работа с моделями.

### Значение для MVP

Это основной переход от уровня workspace к уровню моделирования.

---

## 12.3.3. Просмотреть структуру проекта

### Описание операции

Пользователь просматривает дерево папок, моделей и нотаций в левой панели.

### Результат операции

Пользователь понимает, какие модели уже существуют, как они физически организованы и с чем он может продолжить работу.

### Значение для MVP

Эта операция делает проект обозримым, а не скрытым внутренним контейнером.

---

## 12.3.4. Создать папку внутри проекта

### Описание операции

Пользователь создаёт новую подпапку внутри каталога моделей.

### Результат операции

В дереве проекта появляется новая папка, которую можно использовать для группировки моделей.

### Значение для MVP

Это поддерживает исследовательскую работу с несколькими связанными диаграммами.

---

## 12.4. Операции уровня модели

## 12.4.1. Создать новую модель

### Описание операции

Пользователь создаёт новую модель внутри проекта и выбирает её начальный режим:

* `freeform`;
* `typed`, если хочет сразу выбрать существующую нотацию.

### Результат операции

Система создаёт новый YAML-файл модели и открывает его на canvas.

### Значение для MVP

Это одна из центральных операций продукта, потому что модель — базовая рабочая единица системы.

---

## 12.4.2. Открыть существующую модель

### Описание операции

Пользователь выбирает YAML-файл модели в левом дереве.

### Результат операции

Модель открывается в центральной области редактора.

### Значение для MVP

Это основной способ перемещения между диаграммами.

---

## 12.4.3. Переименовать модель

### Описание операции

Пользователь изменяет отображаемое имя модели через свойства модели.

### Результат операции

Обновляется пользовательское имя модели, при этом внутренняя идентичность и ссылки должны сохраняться.

### Значение для MVP

Эта операция нужна для постепенного прояснения структуры проекта.

---

## 12.4.4. Назначить или изменить нотацию модели

### Описание операции

Пользователь связывает модель с выбранной нотацией или меняет такую связь.

### Результат операции

Модель переходит в режим typed или меняет текущую нотацию.

### Значение для MVP

Это поддерживает переход от свободной карты к более формализованной модели.

---

## 12.4.5. Сохранить модель

### Описание операции

Пользователь сохраняет текущую модель.

### Результат операции

Состояние диаграммы, её элементов, связей, frame, drill-down и step-up записывается в YAML-файл.

### Значение для MVP

Без этой операции редактор не выполняет роль устойчивой песочницы.

---

## 12.4.6. Закрыть модель и снова открыть её

### Описание операции

Пользователь выходит из модели и затем открывает её повторно.

### Результат операции

Модель должна восстановиться без потери структуры.

### Значение для MVP

Эта операция проверяет целостность формата хранения и поведения системы.

---

## 12.5. Операции над canvas и базовыми элементами

## 12.5.1. Создать node

### Описание операции

Пользователь создаёт новый объект на canvas через контекстное меню или иной минимальный механизм добавления.

### Результат операции

На диаграмме появляется новая node с уникальным ID и базовыми начальными свойствами.

### Значение для MVP

Это базовая операция любой содержательной работы на диаграмме.

---

## 12.5.2. Переместить node

### Описание операции

Пользователь перемещает объект по canvas.

### Результат операции

Изменяются координаты node, и новое положение сохраняется как часть модели.

### Значение для MVP

Позволяет пользователю визуально структурировать карту.

---

## 12.5.3. Изменить имя node

### Описание операции

Пользователь редактирует label объекта в панели свойств.

### Результат операции

Node получает новое отображаемое имя без изменения своего ID.

### Значение для MVP

Это ключевая операция исследовательской работы, потому что пользователь часто уточняет смысл объекта уже после его появления.

---

## 12.5.4. Изменить описание node

### Описание операции

Пользователь добавляет или редактирует описание объекта.

### Результат операции

У node появляется более явный текстовый контекст.

### Значение для MVP

Это помогает удерживать смысл и использовать диаграмму не только как картинку, но и как рабочую структуру понятий.

---

## 12.5.5. Удалить node

### Описание операции

Пользователь удаляет объект с диаграммы.

### Результат операции

Удаляется сама node, локальные связи с ней и, при необходимости, система запрашивает подтверждение, если у объекта были drill-down-ссылки.

### Значение для MVP

Это позволяет перестраивать модель без ручного редактирования YAML.

---

## 12.6. Операции над связями

## 12.6.1. Создать edge

### Описание операции

Пользователь соединяет две node направленной связью.

### Результат операции

На диаграмме появляется edge между source и target.

### Значение для MVP

Это базовый способ выразить структуру отношения между объектами.

---

## 12.6.2. Изменить подпись edge

### Описание операции

Пользователь задаёт или меняет текстовую подпись связи.

### Результат операции

Связь получает label, помогающий пояснить её смысл.

### Значение для MVP

Хотя связи ещё нетипизированы, подпись даёт минимальную содержательную семантику.

---

## 12.6.3. Удалить edge

### Описание операции

Пользователь удаляет связь.

### Результат операции

Удаляется только сама edge, без затрагивания node и других элементов.

### Значение для MVP

Это поддерживает гибкую пересборку структуры.

---

## 12.7. Операции над frame

## 12.7.1. Создать frame

### Описание операции

Пользователь выделяет несколько node и объединяет их во frame.

### Результат операции

На диаграмме появляется новый frame с собственным ID и составом включённых объектов.

### Значение для MVP

Это одна из центральных операций, потому что через frame начинается движение к укрупнению структуры.

---

## 12.7.2. Задать имя frame

### Описание операции

Пользователь задаёт frame label.

### Результат операции

Группа объектов начинает восприниматься как отдельный смысловой фрагмент.

### Значение для MVP

Эта операция переводит чистую группировку в осмысленное объединение.

---

## 12.7.3. Задать описание frame

### Описание операции

Пользователь добавляет описание для frame.

### Результат операции

Frame получает дополнительный контекст и становится более устойчивой концептуальной единицей.

### Значение для MVP

Это важно, потому что frame у тебя — не декоративная рамка, а носитель перехода к верхнему уровню.

---

## 12.7.4. Изменить состав frame

### Описание операции

Пользователь добавляет node во frame или убирает их из него.

### Результат операции

Обновляется состав контейнера.

### Значение для MVP

Это позволяет корректировать уже выделенные смысловые фрагменты по мере развития диаграммы.

---

## 12.7.5. Удалить frame

### Описание операции

Пользователь удаляет frame.

### Результат операции

Удаляется только контейнер, а node внутри остаются на диаграмме.

### Значение для MVP

Это сохраняет свободу перестройки без разрушения содержимого модели.

---

## 12.8. Операции drill-down

## 12.8.1. Добавить drill-down к node

### Описание операции

Пользователь выбирает node и добавляет ей детализирующую модель.

### Результат операции

У node появляется одна запись в списке drill-down.

### Значение для MVP

Это одна из ключевых операций, поддерживающих движение вниз по уровню абстракции.

---

## 12.8.2. Создать новую drill-down модель

### Описание операции

Пользователь не просто привязывает существующую модель, а создаёт новую детализирующую модель из панели свойств выбранной node.

### Результат операции

Система создаёт новый YAML-файл модели и автоматически связывает его с текущей node.

### Значение для MVP

Это делает детализацию естественной частью потока работы.

---

## 12.8.3. Привязать существующую модель как drill-down

### Описание операции

Пользователь выбирает уже существующую модель проекта и назначает её детализацией для выбранной node.

### Результат операции

Формируется ссылка от node к выбранной модели.

### Значение для MVP

Это позволяет переиспользовать уже созданные диаграммы в качестве разных представлений объекта.

---

## 12.8.4. Перейти в drill-down модель

### Описание операции

Пользователь открывает связанную детализирующую модель.

### Результат операции

Редактор открывает другую диаграмму, сохраняя понятную связь с исходной node.

### Значение для MVP

Это проверяет главный механизм детализации системы.

---

## 12.8.5. Вернуться из drill-down

### Описание операции

Пользователь возвращается назад из детализирующей модели.

### Результат операции

Редактор возвращает пользователя к исходной диаграмме и исходной точке контекста.

### Значение для MVP

Без этой операции drill-down будет восприниматься как разрыв, а не как часть одной структуры.

---

## 12.8.6. Удалить drill-down ссылку

### Описание операции

Пользователь удаляет связь между node и одной из её детализирующих моделей.

### Результат операции

Удаляется именно ссылка, а не обязательно сама дочерняя модель.

### Значение для MVP

Это важно, чтобы пользователь мог перестраивать сеть моделей без потери уже созданных файлов.

---

## 12.9. Операции step-up

## 12.9.1. Выполнить step-up для frame

### Описание операции

Пользователь выбирает frame и запускает действие step-up.

### Результат операции

Система:

* создаёт новую верхнеуровневую модель;
* создаёт в ней node, представляющую данный frame;
* сохраняет связь между frame и новой моделью.

### Значение для MVP

Это центральная операция укрупнения структуры.

---

## 12.9.2. Перейти в upper-level модель

### Описание операции

Пользователь открывает модель, созданную через step-up.

### Результат операции

Он попадает на более общий уровень описания.

### Значение для MVP

Эта операция делает step-up не абстрактной функцией, а реальным механизмом работы с уровнями.

---

## 12.9.3. Вернуться из upper-level модели к frame

### Описание операции

Пользователь возвращается к исходному frame из верхнеуровневой модели.

### Результат операции

Связь между уровнями сохраняется как двусторонняя.

### Значение для MVP

Это подтверждает, что система поддерживает не одноразовую генерацию, а устойчивую структуру переходов.

---

## 12.9.4. Пересоздать upper-level после изменения frame

### Описание операции

Если состав frame изменился, пользователь может заново выполнить step-up или обновить верхнеуровневую модель вручную.

### Результат операции

Связь между уровнями переопределяется или уточняется.

### Значение для MVP

Автоматическая синхронизация пока не обязательна, но операция явного обновления логически понадобится даже в упрощённой форме.

---

## 12.10. Операции поздней типизации

## 12.10.1. Назначить тип node

### Описание операции

Пользователь выбирает существующую node и присваивает ей тип.

### Результат операции

Node остаётся на месте, но получает более формализованную роль.

### Значение для MVP

Это одна из ключевых операций всей системы, потому что она позволяет переходить от свободной карты к структурированной модели без пересоздания диаграммы.

---

## 12.10.2. Изменить тип node

### Описание операции

Пользователь меняет ранее назначенный тип объекта.

### Результат операции

Node переосмысляется внутри модели, не теряя своей идентичности и связей.

### Значение для MVP

Это важно для исследовательской работы, где типы уточняются постепенно.

---

## 12.10.3. Снять тип с node

### Описание операции

Пользователь убирает тип, возвращая объект к более свободному состоянию.

### Результат операции

Node снова становится нестрого типизированной.

### Значение для MVP

Это поддерживает обратимость формализации и не делает типизацию необратимым решением.

---

## 12.10.4. Перевести модель в typed-режим

### Описание операции

Пользователь связывает модель с нотацией и тем самым делает её typed-моделью.

### Результат операции

Модель получает notation reference, а пользователь может типизировать элементы в рамках выбранной нотации.

### Значение для MVP

Это связывает свободное моделирование с режимом повторного использования возникшего языка.

---

## 12.11. Операции работы с нотациями

## 12.11.1. Создать нотацию из текущей модели

### Описание операции

Пользователь запускает действие создания новой нотации на основе уже прояснённой диаграммы.

### Результат операции

Система:

* извлекает типы;
* формирует YAML-файл нотации;
* сохраняет его в проекте;
* делает нотацию доступной для новых моделей.

### Значение для MVP

Это одна из самых важных операций продукта, потому что именно она переводит практику моделирования в reusable language.

---

## 12.11.2. Связать текущую модель с созданной нотацией

### Описание операции

После создания нотации пользователь может привязать текущую модель к ней.

### Результат операции

Исходная свободная диаграмма становится typed-моделью, не теряя своей структуры.

### Значение для MVP

Это фиксирует переход от свободного режима к формализованному.

---

## 12.11.3. Создать новую модель по существующей нотации

### Описание операции

Пользователь создаёт новую модель и выбирает одну из доступных нотаций проекта.

### Результат операции

Открывается пустая typed-модель, в которой пользователь работает уже в рамках ранее созданного языка типов.

### Значение для MVP

Эта операция показывает, что нотация действительно начинает работать как средство повторного моделирования.

---

## 12.12. Операции навигации и удержания контекста

## 12.12.1. Перейти между моделями через дерево проекта

### Описание операции

Пользователь открывает любую модель напрямую из левой панели.

### Результат операции

Редактор переключается на выбранную диаграмму.

### Значение для MVP

Это даёт независимый от drill-down и step-up способ перемещения по workspace.

---

## 12.12.2. Понять текущий контекст

### Описание операции

Хотя это не “кнопка”, это тоже важная операция: пользователь должен иметь возможность быстро понять, где он находится.

### Результат операции

Пользователь видит:

* текущий проект;
* текущую модель;
* режим модели;
* связь этой модели с предыдущим переходом.

### Значение для MVP

Без этой возможности сложная сеть моделей станет когнитивно неуправляемой.

---

## 12.12.3. Вернуться назад по истории переходов

### Описание операции

Пользователь возвращается к предыдущей модели после переходов по drill-down или step-up.

### Результат операции

Контекст работы не теряется.

### Значение для MVP

Это делает работу с уровнями естественной, а не разорванной.

---

## 12.13. Операции сохранения и устойчивости состояния

## 12.13.1. Сохранить проект после изменений

### Описание операции

Пользователь фиксирует текущее состояние работы.

### Результат операции

Все изменения в моделях, связях и нотациях записаны на диск.

### Значение для MVP

Поддерживает накопительный режим исследования.

---

## 12.13.2. Повторно открыть проект без потери структуры

### Описание операции

Пользователь возвращается к проекту позже.

### Результат операции

Восстанавливаются:

* модели;
* node и edge;
* frame;
* drill-down и step-up;
* типизация;
* нотации.

### Значение для MVP

Это подтверждает, что редактор работает не как временная доска, а как среда длительной работы.

---

## 12.14. Минимальный обязательный набор операций MVP

Хотя полный список выше шире, для проверки центральной идеи MVP критичны следующие операции:

1. Создать проект.
2. Открыть проект.
3. Создать freeform-модель.
4. Создать node.
5. Создать edge.
6. Создать frame.
7. Выполнить step-up.
8. Создать drill-down.
9. Перейти между связанными моделями.
10. Назначить тип node.
11. Создать нотацию из модели.
12. Создать новую typed-модель по этой нотации.
13. Сохранить и заново открыть проект.

Если этот набор работает устойчиво, первая версия уже реализует свою основную продуктовую идею.

---

## 12.15. Итоговый смысл этой главы

В первой версии редактор должен поддерживать не случайный набор команд, а ограниченную систему операций, которая позволяет пользователю постепенно выращивать структуру из наброска. Операции над проектом создают рабочее пространство. Операции над моделью создают отдельные диаграммы. Операции над node, edge и frame позволяют строить карту. Drill-down и step-up создают переходы между уровнями. Поздняя типизация и создание нотации превращают возникающую структуру в язык, который можно использовать повторно.

Именно поэтому операции MVP нужно понимать не как меню функций, а как последовательность действий, поддерживающих основной исследовательский цикл продукта.
