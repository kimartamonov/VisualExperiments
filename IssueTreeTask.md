# IssueTreeTask

## Назначение файла

Этот файл является **мастер-промптом / мастер-инструкцией** для перехода от полного пакета проектной документации к **последовательному Issue Tree**, в котором вся реализация продукта разложена на Milestones и последовательные Issues.

Главная цель этого файла:

- на основе уже созданных документов построить **полный план реализации**;
- оформить этот план как **структуру папок и `.md`-файлов**;
- сделать так, чтобы **каждый Issue был отдельным файлом**;
- обеспечить **строгую последовательность реализации**: Issue за Issue, шаг за шагом;
- сделать каждый Issue достаточно конкретным, чтобы его можно было брать в работу без дополнительных неявных решений;
- сделать так, чтобы после завершения одного Issue было понятно, что именно уже должно работать для старта следующего.

---

## Когда использовать этот файл

Использовать этот файл нужно тогда, когда:

- уже существует полный пакет проектной документации;
- продуктовая идея, MVP, требования, архитектура, delivery и planning уже зафиксированы;
- нужно перейти от документации к реальному plan of execution;
- нужно создать **Issue Tree**, пригодный для реализации человеком или AI;
- требуется не просто список задач, а **последовательная структура реализации**.

Если проект находится только на стадии brief или discovery и еще не имеет стабильных требований и planning-слоя, использовать этот файл преждевременно.

## Главный ожидаемый результат

Нужно создать **Issue Tree** в файловой структуре, где:

- есть общая папка с roadmap и индексами;
- Milestones разложены по отдельным папкам;
- внутри каждого Milestone лежат отдельные `.md`-файлы;
- **каждый `.md`-файл = один Issue**;
- все Issues упорядочены так, чтобы их можно было реализовывать последовательно;
- dependencies между задачами указаны явно;
- каждый Issue имеет собственные acceptance criteria, tests и criteria for handoff;
- в конце всей структуры виден полный путь реализации MVP.

---

## Какая структура должна быть создана

Рекомендуемая структура результата:

```text
Issue-Tree/
  00_Master_Roadmap.md
  01_Milestone_Index.md
  02_Dependency_Map.md
  03_Traceability_Matrix.md

  M1_Project_Foundation/
    README.md
    M1-01_....md
    M1-02_....md
    M1-03_....md

  M2_Freeform_Modeling_Workspace/
    README.md
    M2-01_....md
    M2-02_....md
    M2-03_....md

  M3_Hierarchy_and_Semantic_Navigation/
    README.md
    M3-01_....md
    M3-02_....md

  M4_Typing_and_Notation_Workflow/
    README.md
    M4-01_....md
    M4-02_....md

  M5_Stability_and_Demo_Readiness/
    README.md
    M5-01_....md
    M5-02_....md
```

Можно использовать и другое имя корневой папки, но структура должна сохранять те же принципы:

- общие overview-файлы наверху;
- отдельные milestone folders;
- отдельный markdown на каждый issue;
- числовая сортировка, отражающая порядок реализации.

## Что должно быть в overview-файлах

## 00_Master_Roadmap.md

Этот файл должен содержать:

- краткое описание цели всего Issue Tree;
- список всех Milestones;
- цель каждого Milestone;
- порядок прохождения Milestones;
- общий критерий завершения MVP;
- ключевые dependency gates;
- список critical decisions, которые влияют на реализацию.

Этот файл нужен как главный "roadmap at a glance".

---

## 01_Milestone_Index.md

Этот файл должен содержать:

- полный список Milestones;
- для каждого Milestone:
  - ID;
  - название;
  - цель;
  - capability slices;
  - список issue IDs в порядке реализации;
  - exit criteria.

---

## 02_Dependency_Map.md

Этот файл должен содержать:

- зависимости между Milestones;
- зависимости между critical Issues;
- отдельное выделение blocking issues;
- указание issue, после которого можно начинать следующий большой блок.

Важно: dependencies должны быть описаны **в последовательной форме**, а не как абстрактный граф ради графа.

---

## 03_Traceability_Matrix.md

Этот файл должен связывать:

- Issues;
- Capability Slices;
- FR;
- AC;
- Demo Steps;
- Decision Backlog items;
- Risks.

Минимальная форма — таблица трассировки.

Этот файл нужен, чтобы любой Issue можно было проследить назад до исходной документации.

---

## Что должно быть в каждом Milestone README

Каждая папка Milestone должна содержать `README.md`, в котором изложено:

- ID и название Milestone;
- цель Milestone;
- пользовательский результат Milestone;
- какие capability slices он покрывает;
- какие решения должны быть уже закрыты до старта;
- список issues в порядке реализации;
- что является входом в Milestone;
- что является exit criteria;
- какие риски наиболее важны внутри Milestone;
- какая минимальная валидация закрывает Milestone.

## Главный принцип построения Issue Tree

Issue Tree должен быть **не просто набором задач**, а системой последовательного продвижения.

Это означает:

- каждый следующий Issue опирается на уже завершенный предыдущий контекст;
- если существует blocking decision, он идет раньше implementation;
- если нужен spike, он идет раньше implementation, который от него зависит;
- если capability требует проверки, после implementation идет validation issue;
- bugfix issues появляются как следствие validation, а не как скрытый новый scope.

Главное правило:

> Вся структура должна быть такой, чтобы команда могла идти по ней сверху вниз, практически не ломая голову над тем, "а что делать дальше".

---

## Требования к составу Issues

Issue Tree обязательно должен содержать следующие типы задач:

### 1. Decision Issues

Нужны там, где есть unresolved decisions из `Decision Backlog`.

### 2. Spike Issues

Нужны там, где есть техническая неопределенность, которую дешевле быстро проверить.

### 3. Implementation Issues

Основные задачи на реализацию конкретной capability.

### 4. Validation Issues

Нужны для milestone-level или capability-level проверки результата.

### 5. Bugfix Issues

Нужны для исправления критических и высоких проблем, найденных validation.

### 6. Refinement Issues

Допустимы только после того, как core behavior уже реализован и принят локально.

---

## Главный формат каждого Issue

Каждый Issue должен быть оформлен как отдельный `.md`-файл.

Файл должен содержать **две части**:

1. `Metadata`
2. `Issue body`

---

## Рекомендуемый шаблон metadata

В начале каждого issue-файла должен быть компактный metadata block.

Пример структуры:

```md
# [M2-03] Create and persist nodes in freeform model

## Metadata

- Issue ID: M2-03
- Type: Implementation
- Status: Proposed
- Milestone: M2. Freeform Modeling Workspace
- Capability Slice: CS-04. Node editing
- Priority: P0
- Sequence Order: 3
- Depends On:
  - M2-01
  - M2-02
- Unlocks:
  - M2-04
  - M2-05
- Decision References:
  - DB-12
  - DB-15
- FR References:
  - FR-3.1
  - FR-3.2
  - FR-3.3
  - FR-3.4
  - FR-3.5
- AC References:
  - AC-4
- Demo References:
  - Step 4
- Risk References:
  - R3
```

Если используется другой формат metadata, он все равно должен содержать эти же сущности.

---

## Обязательные разделы внутри каждого Issue

После metadata каждый Issue должен содержать следующие разделы.

### 1. Goal

Коротко и однозначно:

- какой именно результат должен быть достигнут;
- какую capability он добавляет или стабилизирует.

### 2. Why Now

Зачем этот Issue идет именно сейчас:

- какую зависимость он закрывает;
- какой следующий шаг он разблокирует;
- почему его нельзя безопасно отложить.

### 3. User/System Outcome

Что станет возможным после завершения Issue:

- с точки зрения пользователя;
- с точки зрения системы;
- с точки зрения roadmap progression.

### 4. Scope

Что входит в Issue.

Нужно писать списком конкретных ожидаемых результатов.

### 5. Out of Scope

Что **не** входит в этот Issue, чтобы защитить его от расползания.

### 6. Preconditions

Что уже должно работать к моменту начала этого Issue.

Это обязательно должно быть сформулировано как:

- completed issues;
- already working behavior;
- stable decisions;
- required artifacts or contracts.

### 7. Implementation Notes

Не низкоуровневый код, а guidance:

- на какие документы опираться;
- какие слои системы затрагиваются;
- какие ограничения нельзя нарушить;
- на какие решения смотреть при спорных трактовках.

### 8. Files and Artifacts Expected to Change

Нужно перечислить, какие типы артефактов ожидаемо будут затронуты:

- frontend;
- backend;
- API;
- YAML schema;
- docs;
- test assets;
- demo artifacts.

Если конкретные пути пока неизвестны, можно указывать типы артефактов.

### 9. Acceptance Criteria for This Issue

Это **локальные acceptance criteria Issue**, а не только ссылки на глобальные AC.

Здесь нужно перечислить:

- что именно должно работать;
- что пользователь должен мочь сделать;
- что должно сохраняться;
- что не должно ломаться;
- какие ошибки должны обрабатываться, если это релевантно.

### 10. Required Tests

Для каждого Issue обязательно должен быть раздел с тестами / проверками.

Нужно указать, какие проверки должны быть пройдены до закрытия Issue.

Минимально необходимо перечислять:

- `Functional checks`
- `Smoke checks`
- `Regression checks`

А если релевантно, то также:

- `Persistence/reload checks`
- `Navigation checks`
- `Error handling checks`
- `Manual scenario checks`
- `Unit/integration tests`, если они приняты в проекте на этом этапе

### 11. Handoff to Next Issue

Это один из самых важных разделов.

Он должен отвечать на вопрос:

> Что именно теперь считается рабочим и на что может опираться следующий Issue?

Тут нужно перечислить:

- что теперь стабильно работает;
- какой контракт теперь можно считать зафиксированным;
- какой следующий issue или группа issues теперь разблокированы.

### 12. Done Definition

Финальный список того, когда этот Issue можно считать завершенным.

Этот раздел должен быть максимально конкретным.

---

## Обязательные требования к разделу Required Tests

Каждый Issue должен иметь **не пустой** раздел проверок.

Нельзя оставлять "tests TBD".

Нужно явно указать, какие именно проверки нужны.

### Минимальный шаблон tests

Для каждого Issue нужно определить:

#### Functional checks

- основной happy path работает;
- действие доступно через ожидаемый UI/API;
- результат соответствует Goal.

#### Smoke checks

- feature не приводит к крашу;
- базовый цикл использования работает повторяемо.

#### Regression checks

- ранее реализованные capability, от которых зависит этот Issue, не сломаны.

### Дополнительные обязательные проверки по типу Issue

#### Если Issue затрагивает данные или YAML

Нужно добавить:

- сохранение;
- повторное открытие;
- корректность сериализации;
- отсутствие потери структуры.

#### Если Issue затрагивает drill-down / step-up / navigation

Нужно добавить:

- переход туда;
- переход обратно;
- сохранение контекста;
- поведение при битой ссылке, если это релевантно.

#### Если Issue затрагивает UI state и editor behavior

Нужно добавить:

- интерактивность;
- предсказуемость выделения/редактирования;
- отсутствие критических UI failures.

#### Если Issue decision-only

Нужно добавить:

- письменную фиксацию решения;
- обновление соответствующего planning-артефакта;
- отсутствие двусмысленности для следующего implementation issue.

---

## Обязательные требования к разделу Handoff to Next Issue

Каждый Issue обязан явно фиксировать, что считается "готовым основанием" для следующего шага.

Этот раздел должен содержать:

- `What now works`
- `What contract is now stable`
- `What next issue can start`

Без этого Issue Tree теряет главное свойство — последовательность.

---

## Требования к порядку Issues

Issue Tree должен быть организован так, чтобы сохранялась **единая основная последовательность**.

Даже если некоторые задачи теоретически можно было бы делать параллельно, в дереве должен существовать:

- явный `preferred sequential order`;
- один основной путь реализации;
- ясный порядок "после чего делать что".

Нельзя генерировать backlog как плоский набор независимых задач.

---

## Какой уровень детализации должен быть у одного Issue

Один Issue должен соответствовать одному проверяемому инкременту.

### Допустимый размер Issue

Issue должен быть достаточно маленьким, чтобы:

- его можно было принять локально;
- он имел понятный результат;
- его можно было завершить без скрытых product decisions.

### Недопустимый размер Issue

Нельзя делать в одном Issue:

- несколько capability slices сразу;
- большой кусок milestone без промежуточной приемки;
- и decision, и implementation, и validation одновременно, если они реально разделимы.

## Правила naming convention

### Корневая папка

Рекомендуемое имя:

- `Issue-Tree/`

### Папки Milestones

Формат:

- `M1_Project_Foundation`
- `M2_Freeform_Modeling_Workspace`
- `M3_Hierarchy_and_Semantic_Navigation`
- `M4_Typing_and_Notation_Workflow`
- `M5_Stability_and_Demo_Readiness`

### Файлы Issues

Формат:

- `M1-01_Finalize_Project_File_Placement.md`
- `M1-02_Create_Project_Bootstrap.md`
- `M2-03_Create_and_Persist_Nodes.md`

Имя файла должно содержать:

- milestone prefix;
- numeric order;
- короткое действие/результат в ASCII-friendly форме.

---

## Как Issues должны быть сгруппированы по типам внутри sequence

Внутри capability-среза рекомендуемый порядок такой:

1. `Decision`, если есть незакрытая неопределенность
2. `Spike`, если нужен быстрый технический ответ
3. `Implementation`
4. `Validation`
5. `Bugfix`
6. `Refinement`, если действительно нужен

---

## Что обязательно должно быть отражено в Issue Tree

Issue Tree обязан включать:

### 1. Все P0 capability slices

Нельзя пропускать обязательные slices из planning-документов.

### 2. Все blocking decisions

Если решение помечено как critical for decomposition, оно должно появиться в дереве.

### 3. Save/reload integrity

Нельзя оставить сохранение и повторное открытие "на потом", если feature уже меняет data model.

### 4. Validation chain

Каждый milestone должен иметь хотя бы один validation issue.

### 5. Acceptance path

Весь 14-step demo flow должен быть покрыт набором Issues.

### 6. Risk-sensitive checks

Если в risk register есть риск, связанный с feature, в Issue Tree должна быть либо mitigation-task, либо соответствующая проверка.

---

## Чего нельзя допускать в Issue Tree

### 1. Технических bucket-задач без результата

Например:

- "Сделать архитектуру frontend"
- "Подготовить backend систему"

Такие формулировки недопустимы.

### 2. Mega-Issues

Нельзя объединять в один Issue слишком большой кусок поведения.

### 3. Скрытых решений

Если Step-up semantics еще не выбрана, нельзя писать implementation issue так, будто все уже решено.

### 4. Отрыва от документации

Issue без FR/AC/demo/decision references не должен существовать.

### 5. Пустых sections

Нельзя оставлять пустыми:

- tests;
- acceptance;
- preconditions;
- handoff to next issue.

---

## Главный критерий качества Issue Tree

Хороший Issue Tree должен отвечать "да" на следующие вопросы:

1. Можно ли пройти реализацию сверху вниз, почти не останавливаясь на доп. декомпозицию?
2. Понятно ли, зачем существует каждый Issue?
3. Понятно ли, почему он идет именно сейчас?
4. Понятно ли, как его принять?
5. Понятно ли, что будет основанием для следующего Issue?
6. Покрывает ли дерево весь MVP, а не только отдельные features?
7. Есть ли у каждого Milestone четкая точка выхода?

---

## Короткий шаблон задачи для генератора

Ниже краткая формулировка самой задачи, которую можно использовать как operational core этого мастер-промпта:

> На основе полного пакета проектной документации создай `Issue-Tree/` как файловую структуру Milestones и последовательных Issues. Каждый Issue должен быть отдельным `.md`-файлом. Сгенерируй общий roadmap, milestone index, dependency map и traceability matrix. Внутри каждого Milestone создай `README.md` и упорядоченный набор issue-файлов. Каждый Issue должен содержать metadata, goal, why now, scope, out of scope, preconditions, implementation notes, expected artifacts, local acceptance criteria, required tests, handoff to next issue и done definition. Все Issues должны быть расположены в явной последовательности, пригодной для пошаговой реализации без скрытых product decisions.

---

## Рекомендуемая reusable формулировка для другого проекта

Если этот файл используется в другом проекте, рекомендованная команда может звучать так:

> Используя данные из `docs`, реализуй написание `Issue Tree`, пользуясь инструкцией `IssueTreeTask.md`, в которой описаны структура результата, требования к Milestones, правила последовательности, формат файлов и обязательное содержание каждого Issue. Создай результат не в виде простого ответа в чате, а в виде файловой структуры `Issue-Tree/`, где каждый Issue представлен отдельным `.md`-файлом.

Можно использовать и более полную версию:

> Используя данные из `docs`, создай полный `Issue Tree` по инструкции из `IssueTreeTask.md`. Построй результат как файловую структуру `Issue-Tree/` с overview-файлами, milestone folders и отдельными `.md`-файлами для каждого Issue. Все Issues должны быть упорядочены в последовательности реализации, иметь явные зависимости, содержать acceptance criteria, required tests, handoff to next issue и done definition. Дерево должно покрывать весь MVP, все P0 capability slices, blocking decisions, milestone validation и полный путь к прохождению demo scenario.

---

## Что должен сделать исполнитель по этой команде

Если генератор получает команду в форме:

> Используя данные из `docs`, реализуй написание `Issue Tree`, пользуясь этой инструкцией `IssueTreeTask.md`

то от него ожидается, что он:

1. Считает `docs/` основным source of truth.
2. Использует `IssueTreeTask.md` как master-instruction.
3. Создает именно файловую структуру, а не только текстовый summary.
4. Строит Milestones на основе capability slices и delivery plan.
5. Строит каждый Issue как отдельный `.md`-файл.
6. Делает строгую последовательность выполнения.
7. Добавляет acceptance, tests и handoff для каждого Issue.
8. Проверяет, что по Issue Tree можно пройти от первого шага до готового MVP.

---

## Короткая проверка готовности IssueTreeTask.md

`IssueTreeTask.md` можно считать готовым для переноса в другой проект, если он:

- не привязан к конкретному домену текущего продукта;
- ссылается на `docs` как на входной пакет;
- задает структуру выходных файлов;
- задает обязательный формат каждого Issue;
- требует последовательности, а не просто списка задач;
- требует tests, acceptance и handoff;
- допускает использование как standalone master prompt.

Именно в таком виде его и следует использовать в другом репозитории.

---

## Финальное правило

Главная задача этого файла — обеспечить, чтобы реализация продукта была представлена не как "список тем", а как **жестко структурированная последовательность Issues**, где:

- каждый шаг опирается на предыдущий;
- каждый шаг имеет понятную приемку;
- каждый шаг имеет обязательные проверки;
- каждый шаг явно готовит следующий;
- весь путь от первого Issue до готового MVP виден заранее.

Именно в этом состоит смысл `IssueTreeTask.md`.
