Собираю это в цельный документ.

Твоя идея в текущем виде — это **не просто редактор диаграмм**, а **sandbox для рождения нотаций и моделей из практики рисования**.
Ключевой замысел такой: пользователь может сначала мыслить свободной схемой, потом различать в ней устойчивые типы, фиксировать их как нотацию, а затем уже работать в рамках этой нотации. Плюс редактор должен поддерживать движение **вниз** по детализации и **вверх** по абстракции.

---

# Brief v0.1

## Проект: визуальный редактор моделей и нотаций

## 1. Краткое описание

Нужно разработать веб-приложение, запускаемое на сервере, которое позволяет:

* работать с проектами как с наборами файлов;
* открывать и редактировать визуальные модели;
* создавать модели как в **свободном режиме** (`freeform`), так и по заранее определённой **нотации**;
* выполнять **drill-down**: раскрывать объект в дочернюю модель;
* выполнять **step-up**: сворачивать группу объектов в более верхнеуровневую модель;
* извлекать из свободной диаграммы новую **нотацию**.

Первая версия — это **песочница для экспериментов с верхнеуровневыми фреймворками**, а не production-редактор с жёсткой формализацией.

---

## 2. Цель MVP

Сделать минимально рабочую среду, в которой можно:

1. создать проект;
2. открыть визуальный редактор;
3. нарисовать свободную диаграмму;
4. добавить объекты, стрелки и frame;
5. сделать `drill-down` и `step-up`;
6. назначить типы объектам;
7. извлечь из диаграммы новую YAML-нотацию;
8. создать новую диаграмму по этой нотации.

### Главный критерий успеха MVP

Если пользователь может пройти весь этот путь от **freeform-карты** до **собственной нотации** и обратно к **типизированной модели**, MVP считается успешным.

---

## 3. Продуктовая идея

Редактор должен поддерживать два естественных режима работы:

### Режим 1. Моделирование по готовой нотации

Пользователь выбирает существующую YAML-нотацию и создаёт модель в рамках определённых типов объектов.

### Режим 2. Моделирование до нотации

Пользователь сначала рисует свободную диаграмму, а потом:

* назначает типы объектам;
* фиксирует эти типы;
* сохраняет их как новую нотацию.

### Концептуальная особенность

Система поддерживает не только переход:

**нотация → модель**

но и обратный переход:

**свободная модель → нотация**

---

## 4. Целевая аудитория первой версии

Первая версия ориентирована на **одного основного пользователя** — автора фреймворков и экспериментальных моделей.

Это не multi-user платформа и не корпоративный редактор с совместной работой.
MVP делается как личная рабочая песочница.

---

## 5. Границы MVP

## Что входит

* список проектов;
* создание нового проекта;
* вход в проект;
* отображение структуры папок и файлов моделей;
* визуальный редактор на базе React Flow;
* freeform-модели;
* модели по нотации;
* YAML-файлы нотаций, читаемые с диска;
* свойства объектов справа;
* directed edges без типизации;
* frame;
* drill-down;
* step-up;
* создание нотации из существующей диаграммы;
* сохранение моделей в YAML.

## Что не входит

* типизированные связи;
* визуальный редактор метамоделей / нотаций;
* валидация модели;
* совместная работа нескольких пользователей;
* история изменений;
* роли и права доступа;
* компиляция в Windmill;
* сквозные инстансы объектов между диаграммами;
* сложные shape/иконки;
* сложная палитра элементов;
* rich styling beyond basic colors.

---

## 6. Общая архитектурная идея

## Технологическое направление

* Frontend: TypeScript + React
* Canvas: React Flow
* Backend: минимальный сервер для отдачи проектов и файлов
* Хранение: файловая структура на сервере
* Формат моделей и нотаций: YAML

## Важный принцип

На первом этапе YAML-нотации **не редактируются в UI**.
Они создаются автоматически из диаграмм и могут редактироваться вручную во внешнем редакторе, например VS Code.

---

## 7. Основные сущности системы

## 7.1. Project

Проект — это папка с файлами моделей и manifest-файлом.

### Назначение

* объединяет модели;
* задаёт контекст проекта;
* является верхним уровнем навигации.

---

## 7.2. Manifest

Manifest — файл проекта с базовой служебной информацией.

### Минимально должен содержать

* имя проекта;
* список/ссылки на модели проекта;
* стартовые настройки проекта;
* задел под список нотаций.

### Пока можно держать минимальным

Поддержка нотаций в manifest может быть пока упрощённой, но файл должен существовать как точка расширения.

---

## 7.3. Model

Модель — YAML-файл, который хранит диаграмму.

### Модель может быть:

* `freeform`
* или `typed` / связанной с нотацией

### Модель хранит

* идентификатор модели;
* имя;
* тип модели / notation reference;
* список нод;
* список стрелок;
* список frame;
* ссылки на drill-down;
* служебные данные для UI.

---

## 7.4. Notation

Нотация — YAML-файл, определяющий набор типов объектов для использования в модели.

### В MVP нотация описывает

* имя нотации;
* список типов объектов;
* цвет каждого типа;
* задел под рекомендованные нотации для детализации.

Нотации в первой версии не являются самостоятельным визуальным миром.

---

## 7.5. Node

Node — базовый объект диаграммы.

### Минимальные свойства

* стабильный `id` внутри модели;
* `label`;
* `description`;
* `type` или отсутствие типа;
* координаты;
* входящие и исходящие связи;
* список детализирующих моделей.

---

## 7.6. Edge

Edge — направленная стрелка между двумя нодами.

### В MVP

* нет типизации;
* есть только направление;
* хранится по `sourceId` и `targetId`.

---

## 7.7. Frame

Frame — контейнер, выделяющий группу объектов и выражающий их принадлежность к более верхнему объекту.

### Семантика frame

Frame — это не просто декоративная рамка.
Он означает: **эти элементы вместе могут быть свернуты в верхнеуровневый объект**.

### Минимальные свойства frame

* `id`
* `name`
* `description`
* список входящих нод
* возможность выполнить `step-up`

---

## 8. Ключевые операции

## 8.1. Создание проекта

Пользователь может:

* открыть список проектов;
* создать новый пустой проект;
* войти в существующий проект.

---

## 8.2. Работа с файловым деревом

Слева показывается структура папок и YAML-файлов моделей внутри проекта.

### Требование

Показывать именно файловую структуру проекта, а не только абстрактный список моделей.

---

## 8.3. Открытие модели

Пользователь выбирает модель в дереве и открывает её в редакторе.

---

## 8.4. Создание freeform-модели

Пользователь может создать новую модель без нотации.

### Это полноценный режим

`freeform` — не исключение и не временный режим, а первый класс системы.

---

## 8.5. Создание typed-модели

Пользователь может создать модель, основанную на существующей нотации.

---

## 8.6. Редактирование диаграммы

На canvas пользователь может:

* создавать ноды;
* перемещать ноды;
* соединять их стрелками;
* выделять объекты;
* создавать frame;
* редактировать свойства объектов.

---

## 8.7. Поздняя типизация

Пользователь может сначала нарисовать свободную диаграмму, а затем:

* назначить типы объектам;
* присвоить диаграмме смысл;
* сохранить типы как новую нотацию.

### Это центральный сценарий MVP

Поздняя типизация — одна из ключевых продуктовых функций первой версии.

---

## 8.8. Drill-down

Любая нода может быть детализирована в одну или несколько дочерних моделей.

### Управление drill-down

В панели свойств справа пользователь видит список моделей, в которые детализируется выбранная нода, и может:

* открыть существующую;
* добавить новую;
* привязать дополнительную модель.

### Ограничение MVP

Детализирующие модели могут быть как `freeform`, так и связанными с нотацией.

---

## 8.9. Step-up

Пользователь может создать frame, а затем выполнить `step-up`.

### Что делает step-up

* автоматически создаёт новую модель;
* автоматически создаёт в ней ноду, представляющую данный frame;
* связывает frame и верхнеуровневую ноду.

### Смысл

Это обратное движение по отношению к drill-down: не раскрытие объекта, а сворачивание части структуры в более общий объект.

---

## 8.10. Создание нотации из диаграммы

Пользователь может из текущей модели создать новую нотацию.

### Операция должна:

* извлечь список типов из текущей диаграммы;
* сохранить названия типов;
* сохранить цвета типов;
* создать новый YAML-файл нотации;
* пометить текущую диаграмму как модель, относящуюся к этой нотации.

### Важно

Исходная диаграмма не должна теряться или пересоздаваться с нуля.

---

## 9. Пользовательский интерфейс

## Общая компоновка

Интерфейс ориентирован на структуру, похожую по духу на Sirius Web:

### Слева

* список проектов или дерево проекта;
* структура папок;
* список моделей;
* возможность открыть нужный файл.

### В центре

* canvas на React Flow;
* работа с объектами и связями.

### Справа

* панель свойств выделенного объекта;
* если объект не выбран — свойства текущей модели;
* секции панели могут быть сворачиваемыми.

---

## 10. Поведение интерфейса

## Создание объектов

Пока можно использовать упрощённый способ:

* правый клик по canvas;
* контекстное меню `Create new`;
* выбор типа объекта или свободного объекта.

## Внешний вид объектов

В первой версии все объекты могут быть стилистически близки к FigJam:

* прямоугольники со скруглёнными углами;
* простой фон;
* минимальная визуальная вариативность через цвет.

## Типы связей

Пока отсутствуют.
Стрелки только направленные.

## 11. Структура данных

## 11.1. Адресация

У каждой ноды есть стабильный `id` внутри модели.

### Все ссылки строятся по принципу

`relativeModelPath + nodeId`

### Важно

* переименование label не меняет `id`;
* drill-down и связи идут по идентификаторам, а не по имени.

---

## 11.2. Root vs child models

У проекта нет обязательной одной главной модели.
На верхнем уровне может быть несколько независимых моделей.

Модель становится дочерней, когда:

* на неё ссылается нода через drill-down;
* или она создана через step-up / иерархическую связь.

---

## 11.3. Freeform

Свободная диаграмма — это полноценная модель.

### Рекомендуемое техническое представление

`notation: freeform`

Так проще, чем отсутствие поля вообще.

## 13. Черновой формат manifest

```yaml
id: project-001
name: My Project
defaultModel: models/main.yaml
notations:
  - notations/workflow-map.yaml
  - notations/people-needs.yaml
```

---

## 14. Черновой формат нотации

```yaml
id: workflow-map
name: Workflow Map
types:
  - id: actor
    name: Actor
    color: "#F5C26B"
  - id: need
    name: Need
    color: "#8ED1FC"
  - id: process
    name: Process
    color: "#C3F0CA"
recommendedDrilldowns:
  actor:
    - freeform
  process:
    - freeform
```

### Примечание

`recommendedDrilldowns` можно считать необязательным полем MVP, но полезно оставить его как задел.

---

## 15. Черновой формат модели

```yaml
id: model-001
name: Main Map
notation: freeform

nodes:
  - id: n1
    label: User
    description: ""
    type: actor
    position:
      x: 120
      y: 180
    drilldowns:
      - models/research/user-details.yaml

  - id: n2
    label: Need
    description: ""
    type: need
    position:
      x: 420
      y: 180
    drilldowns: []

edges:
  - id: e1
    source: n1
    target: n2

frames:
  - id: f1
    name: User Context
    description: ""
    nodes:
      - n1
      - n2
    stepUp:
      model: models/abstractions/context-map.yaml
      nodeId: n10
```

## 16. Основные сценарии пользователя

## Сценарий 1. Свободное моделирование

1. Пользователь создаёт проект.
2. Заходит в проект.
3. Создаёт новую `freeform`-модель.
4. Рисует несколько объектов.
5. Соединяет их стрелками.
6. Сохраняет модель.

---

## Сценарий 2. Drill-down

1. Пользователь выбирает ноду.
2. В панели свойств добавляет детализирующую модель.
3. Открывает её.
4. Строит более детальную схему.
5. Возвращается назад.

---

## Сценарий 3. Step-up

1. Пользователь выделяет группу объектов во frame.
2. Даёт frame имя.
3. Запускает `step-up`.
4. Система создаёт новую верхнеуровневую модель.
5. В ней появляется объект, представляющий этот frame.
6. Пользователь продолжает строить карту более высокого уровня.

---

## Сценарий 4. Поздняя типизация

1. Пользователь рисует свободную диаграмму.
2. Назначает типы объектам.
3. Нажимает `Create notation`.
4. Система извлекает типы и цвета.
5. Создаёт новую YAML-нотацию.
6. Текущая модель становится связанной с этой нотацией.
7. Пользователь создаёт новую модель уже по этой нотации.

---

## 17. Демонстрационный сценарий MVP

Это основной сквозной сценарий, который должен работать целиком:

1. Создать пустой проект.
2. Открыть freeform-диаграмму.
3. Нарисовать несколько объектов и стрелок.
4. Выделить часть объектов во frame.
5. Выполнить `step-up` и получить верхнеуровневую модель.
6. Вернуться обратно через `dig-down`.
7. Для одной из нод создать отдельный `drill-down`.
8. Назначить типы объектам исходной диаграммы.
9. Сохранить эти типы как новую нотацию.
10. Создать новую диаграмму по этой нотации.

---

## 18. Нефункциональные требования MVP

* приложение должно запускаться на сервере;
* работа должна происходить через веб-интерфейс;
* проект должен быть файлово-ориентированным;
* YAML должен быть основным форматом хранения;
* интерфейс должен быть простым, без избыточной логики;
* кодовая база должна допускать последующее расширение в сторону генерации Windmill flows.

---

## 19. Открытые вопросы

Это не блокеры для старта, а точки будущего уточнения.

### 1. Manifest

Нужно ли в первой версии хранить в manifest полный индекс моделей или достаточно имени проекта и стартовой модели.

### 2. Двойное дерево

Как именно показывать одновременно:

* файловую структуру;
* логическую иерархию drill-down / step-up.

### 3. Frame как сущность

Пока frame — это особый контейнер, а не обычный typed object. Возможно, позже это надо будет формализовать глубже.

### 4. Автоматизация типизации

Сейчас создание нотации предполагает, что типы уже вручную присвоены. В будущем можно добавить полуавтоматический режим.

### 5. Step-up между несколькими frame

Сейчас предполагается один верхний контекст на frame. Позже можно вернуться к более сложным композициям.

---

## 20. Итоговая формулировка для команды

Нужно разработать первую версию веб-редактора моделей, в котором пользователь может работать как со свободными диаграммами, так и с диаграммами по YAML-нотациям. Ключевая особенность продукта — поддержка двух направлений движения: **вниз по детализации** (`drill-down`) и **вверх по абстракции** (`step-up`), а также возможность **извлекать новую нотацию из уже нарисованной свободной диаграммы**.
