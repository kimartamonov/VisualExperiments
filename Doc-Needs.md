# Doc-Needs: Анализ готовности к созданию документов

## Методика

Каждый документ из Doc-Structure оценён по степени покрытия информацией из Brief.txt и 15 файлов Detailed Brief.
- **Готов** — информации в брифе достаточно для создания полноценного документа.
- **Частично готов** — основа есть, но отдельные разделы останутся пустыми или поверхностными.
- **Не готов** — критически не хватает входных данных; документ будет формальной заглушкой.

## Документы, готовые к созданию

### 01. Project Brief
Бриф содержит: описание продукта, идею, бизнес-проблему, цель, ожидаемый результат, целевую аудиторию, критерии успеха, известные ограничения, открытые вопросы. Главы 1–4 Detailed Brief полностью покрывают все разделы.

### 02. Scope and Boundaries
Глава 5 Detailed Brief содержит исчерпывающие списки «что входит» и «что не входит» в MVP, предположения, зависимости, ограничения по технологиям. Brief.txt дублирует эту информацию в разделах 5 и 16.

### 03. Stakeholders and Users
Глава 3 Detailed Brief детально описывает единственного пользователя MVP, характер его работы, контекст использования, уровень подготовки, главную потребность, ограничения. Для MVP с одним пользователем этого достаточно.

### 04. User Scenarios and Use Cases
Глава 6 Detailed Brief содержит 15 пользовательских сценариев. Глава 13 — целостный демонстрационный сценарий из 14 шагов. Brief.txt содержит 4 ключевых сценария и сквозной демо-сценарий. Покрытие полное.

### 05. Functional Requirements
Глава 12 Detailed Brief содержит полный перечень операций по уровням (проект, модель, canvas, связи, frame, drill-down, step-up, типизация, нотации, навигация, сохранение). Главы 8–9 описывают правила поведения сущностей. Из этих данных функциональные требования выводятся напрямую.

### 07. Domain Model
Глава 8 Detailed Brief описывает 10 сущностей с их свойствами, смыслом и связями. Глава 11 содержит YAML-структуры с примерами. Глава 9 фиксирует правила и инварианты. Достаточно для полноценной доменной модели MVP.

### 06. Nonfunctional Requirements

**Краткое описание проблемы:**
В брифе нефункциональные требования описаны одним коротким абзацем (раздел 18 Brief.txt): «запускается на сервере, веб-интерфейс, файлово-ориентированный, YAML, простой UI, расширяемость в сторону Windmill». Это декларации, а не измеримые требования. Нет ни одной количественной метрики.

**На какие вопросы мы не можем ответить:**
- Какова целевая производительность (время отклика, размер модели, количество нод на canvas)?
- Какие браузеры и устройства должны поддерживаться?
- Есть ли требования к безопасности, аутентификации, шифрованию данных?
- Каковы ограничения по размеру проекта, количеству файлов, глубине вложенности?
- Каковы требования к доступности (uptime, recovery)?
- Есть ли требования к логированию и мониторингу?
- Какое максимально допустимое время загрузки проекта / модели?

**Необходимые артефакты:**
- Целевые метрики производительности (время открытия модели, максимальное кол-во нод)
- Список поддерживаемых браузеров / платформ
- Требования к безопасности (даже для однопользовательского режима)
- Ограничения по объёму данных (размер YAML, глубина drill-down)
- Требования к устойчивости работы (что происходит при сбое сервера, потере связи)

**5 вопросов к клиенту:**
1. С каким максимальным количеством объектов на одной диаграмме вы планируете работать (10, 50, 200+)? Это влияет на выбор рендеринга и оптимизации React Flow.
2. На каких устройствах и в каких браузерах вы будете использовать редактор? Нужна ли поддержка планшетов или мобильных?
3. Нужна ли какая-либо аутентификация для доступа к приложению, или оно работает полностью локально / в доверенной сети?
4. Какое поведение ожидается, если сервер упадёт во время работы — достаточно ли ручного перезапуска, или нужно автосохранение / recovery?
5. Есть ли ограничения на размер проекта (количество моделей, глубина вложенности drill-down), которые вы считаете разумными для первой версии?

### 08. System Context and Architecture

**Краткое описание проблемы:**
Глава 14 Detailed Brief фиксирует технологический стек (React, React Flow, Node.js, YAML) и называет 5 архитектурных слоёв, но не раскрывает их внутреннюю структуру, взаимодействие, API-контракты и модель развёртывания. Для полноценного архитектурного документа не хватает решений по конкретным вопросам реализации.

**На какие вопросы мы не можем ответить:**
- Как именно frontend общается с backend (REST, WebSocket, GraphQL)?
- Какова структура API-эндпоинтов?
- Как приложение развёртывается (Docker, bare metal, cloud)?
- Как устроено управление состоянием на frontend (Redux, Zustand, Context)?
- Есть ли reverse proxy, как обслуживается статика?
- Как именно React Flow интегрируется с доменной моделью?

**Необходимые артефакты:**
- Решение по протоколу и формату API (REST endpoints, формат ответов)
- Решение по state management на frontend
- Схема развёртывания (как запускается приложение)
- Решение по структуре модулей frontend и backend
- Описание взаимодействия между слоями (Model Semantics ↔ Persistence ↔ Diagram Engine)

**5 вопросов к клиенту:**
1. Как вы планируете запускать приложение — через Docker, через `npm start` локально, или есть требования к cloud-деплою?
2. Есть ли предпочтения по формату API между frontend и backend (REST / GraphQL / tRPC), или это полностью на усмотрение команды?
3. Нужно ли приложению работать offline (без постоянного соединения с сервером), или постоянная связь с backend гарантирована?
4. Есть ли предпочтения по state management (Redux, Zustand, MobX), или важно только, чтобы архитектура была расширяемой?
5. Планируете ли вы в ближайшем будущем добавлять к backend базу данных или другие сервисы, которые стоит учесть в архитектуре сейчас?

### 09. Data and Integrations

**Краткое описание проблемы:**
Форматы YAML-файлов описаны в главе 11, файловая структура — в главе 10. Однако отсутствует описание API-контрактов между frontend и backend, нет спецификации жизненного цикла данных, нет описания обработки ошибок при чтении/записи файлов, нет сценариев миграции данных при изменении формата.

**На какие вопросы мы не можем ответить:**
- Какие API-эндпоинты нужны для CRUD-операций над проектами, моделями, нотациями?
- Что происходит при конфликте данных (ручное редактирование YAML + одновременная работа в UI)?
- Как будет происходить миграция YAML-формата при обновлении версии?
- Нужна ли валидация YAML при чтении (что делать с некорректными файлами)?
- Как обрабатываются битые ссылки (drill-down на несуществующую модель)?

**Необходимые артефакты:**
- Спецификация API-эндпоинтов (хотя бы список с методами и параметрами)
- Правила валидации YAML при загрузке
- Стратегия обработки ошибок (битые ссылки, невалидный YAML, отсутствующие файлы)
- Решение по конфликтам между ручным и визуальным редактированием
- Стратегия версионирования формата данных

**5 вопросов к клиенту:**
1. Если вы отредактируете YAML-файл вручную (например, в VS Code) с ошибкой, как должен вести себя редактор — показать ошибку, попытаться частично загрузить, или отказать в открытии?
2. Планируете ли вы одновременно редактировать один и тот же файл в VS Code и в визуальном редакторе? Нужно ли автообнаружение внешних изменений?
3. Если drill-down ссылается на модель, которая была удалена или перемещена вручную — какое поведение ожидается?
4. Нужно ли при изменении YAML-формата в будущих версиях автоматически мигрировать старые файлы, или допустимо ручное обновление?
5. Есть ли сценарии экспорта / импорта проектов (архивом, через git), которые стоит учесть в формате хранения?

### 10. Roadmap and Delivery Plan

**Краткое описание проблемы:**
Бриф не содержит никакой информации о планировании разработки: нет сроков, нет этапов, нет команды, нет бюджета, нет приоритетов между функциями внутри MVP. Описан только целевой состав MVP как единое целое, без декомпозиции на итерации.

**На какие вопросы мы не можем ответить:**
- Каков желаемый срок готовности MVP?
- Какова команда разработки (размер, роли, доступность)?
- Каков бюджет проекта?
- В каком порядке реализовывать функции (что важнее первым — drill-down или поздняя типизация)?
- Есть ли внешние дедлайны или зависимости?
- Планируется ли промежуточный релиз до полного MVP?
- Какова методология разработки (спринты, канбан, свободный ритм)?

**Необходимые артефакты:**
- Желаемые сроки (хотя бы ориентировочные)
- Состав и доступность команды
- Приоритизация функций внутри MVP (что можно отложить, если не укладываемся)
- Решение о промежуточных этапах / milestone'ах
- Бюджетные ограничения (если влияют на scope)

**5 вопросов к клиенту:**
1. К какому сроку вы хотели бы получить работающий MVP (месяц, квартал, полугодие)?
2. Кто будет разрабатывать продукт — какой состав команды и сколько часов в неделю они могут посвящать проекту?
3. Если MVP не удаётся реализовать целиком к сроку, какие функции вы готовы отложить на следующую итерацию? Например: step-up, поздняя типизация, создание нотации из модели — что из этого можно сдвинуть?
4. Нужен ли промежуточный результат (например, «через 2 недели хочу видеть canvas с нодами и стрелками»), или вы готовы ждать цельного MVP?
5. Есть ли внешние факторы, которые влияют на сроки (демонстрация инвесторам, начало использования в другом проекте, конференция)?

### 11. Acceptance Criteria and Test Strategy

**Краткое описание проблемы:**
Глава 13 Detailed Brief описывает один демонстрационный сценарий и общий критерий успеха MVP. Однако нет формальных критериев приёмки для каждой отдельной функции, нет стратегии тестирования, нет описания тестовых сред, нет определения «Definition of Done» для задач разработки.

**На какие вопросы мы не можем ответить:**
- Каковы критерии приёмки для каждой из 13 обязательных операций MVP?
- Кто будет принимать результат (сам заказчик, тестировщик)?
- Нужны ли автоматизированные тесты, или достаточно ручной проверки?
- Какие типы тестирования обязательны (unit, integration, e2e)?
- Какие тестовые данные нужны (сколько моделей, какого размера)?
- Что считается критическим багом, а что — допустимой недоработкой MVP?

**Необходимые артефакты:**
- Критерии приёмки для каждой ключевой операции (создание проекта, drill-down, step-up, создание нотации и т.д.)
- Решение о необходимости автоматических тестов в MVP
- Definition of Done для задач разработки
- Приоритизация багов (что блокирует приёмку, что нет)
- Описание процесса приёмки (кто, как, когда проверяет)

**5 вопросов к клиенту:**
1. Как вы планируете принимать результат — будете проходить демо-сценарий самостоятельно, или достаточно видеозаписи прохождения?
2. Нужны ли автоматизированные тесты в первой версии, или для MVP достаточно ручной проверки по сценарию?
3. Если drill-down работает, но step-up имеет мелкие визуальные дефекты — это блокирует приёмку MVP или допустимо?
4. Есть ли у вас набор конкретных моделей / фреймворков, на которых вы хотите проверить MVP (реальные use cases для тестирования)?
5. Что для вас «сломанный MVP» — невозможность пройти весь демо-сценарий целиком, или невозможность отдельной операции?

### 12. Risks, Decisions, Open Questions

**Краткое описание проблемы:**
Глава 15 Detailed Brief содержит хороший список открытых вопросов (по MVP-логике, формату данных, интерфейсу, развитию). Однако нет формальной оценки рисков (вероятность, влияние, митигация), нет журнала архитектурных решений (ADR), нет явных гипотез, требующих проверки, помимо главной.

**На какие вопросы мы не можем ответить:**
- Какие технические риски наиболее критичны (React Flow ограничения, производительность YAML, масштабируемость файловой структуры)?
- Какие принятые решения являются осознанным выбором, а какие — временным компромиссом?
- Какие гипотезы помимо главной нуждаются в проверке?
- Каков план митигации ключевых рисков?
- Кто отвечает за принятие решений по открытым вопросам и в какие сроки?

**Необходимые артефакты:**
- Оценка рисков с вероятностью и влиянием
- Стратегии митигации для топ-5 рисков
- Журнал архитектурных решений (ADR) — хотя бы для ключевых выборов (React Flow, YAML, файловый backend)
- Список гипотез для проверки с критериями валидации
- Ответственные и сроки по открытым вопросам

**5 вопросов к клиенту:**
1. Проводили ли вы прототипирование с React Flow — уверены ли вы, что он поддерживает нужные операции (frame, вложенность, step-up) без серьёзных доработок?
2. Какие из открытых вопросов главы 15 вы считаете критичными для старта разработки, а какие можно решать по ходу?
3. Если в процессе разработки выяснится, что YAML неудобен для хранения больших моделей — готовы ли вы к переходу на другой формат (JSON, SQLite)?
4. Есть ли «красные линии» — решения, которые точно не должны меняться в процессе разработки (например, React Flow, TypeScript, файловое хранение)?
5. Кто принимает финальные решения по спорным архитектурным вопросам — вы лично, или есть технический лидер / консультант?
