# 05. Functional Requirements

## Структура требований

Требования организованы по уровням операций: проект → модель → содержимое модели (canvas) → межуровневые операции → типизация и нотации → навигация → сохранение.

Приоритеты: **P0** — обязателен для MVP, **P1** — желателен для MVP, **P2** — задел.

---

## FR-1. Операции уровня проекта

### FR-1.1. Создать проект [P0]
- Система должна позволить пользователю создать новый проект, указав имя.
- При создании система должна: создать папку проекта, файл `project.yaml` (manifest), каталог `models/`.
- После создания пользователь автоматически попадает в рабочее пространство проекта.

### FR-1.2. Открыть проект [P0]
- Система должна отображать список доступных проектов.
- Пользователь должен иметь возможность выбрать и открыть существующий проект.

### FR-1.3. Просмотреть структуру проекта [P0]
- В левой панели система должна отображать файловую структуру проекта: папки и YAML-файлы моделей.

### FR-1.4. Создать папку внутри проекта [P1]
- Пользователь должен иметь возможность создать подпапку в дереве проекта для организации моделей.

---

## FR-2. Операции уровня модели

### FR-2.1. Создать freeform-модель [P0]
- Система должна позволить создать новую модель без выбора нотации.
- В YAML-файле модели поле `notation` должно быть установлено в `freeform`.

### FR-2.2. Создать typed-модель [P0]
- Система должна позволить создать новую модель с выбором существующей нотации.
- Типы объектов из нотации должны быть доступны при создании нод.

### FR-2.3. Открыть модель [P0]
- Пользователь должен иметь возможность выбрать модель в дереве проекта и открыть её в canvas.

### FR-2.4. Переименовать модель [P1]
- Пользователь должен иметь возможность изменить имя модели.
- Переименование label не должно менять id модели.

### FR-2.5. Назначить / изменить нотацию модели [P1]
- Пользователь должен иметь возможность связать существующую модель с нотацией или изменить привязку.

### FR-2.6. Сохранить модель [P0]
- Система должна сохранять текущее состояние модели в YAML-файл по команде пользователя.

### FR-2.7. Автосохранение [P1]
- Система должна периодически сохранять изменения в фоне без заметных задержек на canvas.
- При отсутствии автосохранения пользователь сохраняет вручную.

---

## FR-3. Операции над canvas и нодами

### FR-3.1. Создать node [P0]
- Пользователь должен иметь возможность создать новый объект на canvas.
- Способ создания: правый клик → контекстное меню → «Create new» → выбор типа (или свободного объекта в freeform-режиме).
- При создании ноде автоматически присваивается стабильный `id`.

### FR-3.2. Переместить node [P0]
- Пользователь должен иметь возможность перемещать ноды на canvas drag-and-drop.
- Координаты ноды должны обновляться и сохраняться.

### FR-3.3. Изменить label node [P0]
- Пользователь должен иметь возможность изменить отображаемое имя ноды.
- Изменение label не должно менять id ноды.

### FR-3.4. Изменить description node [P1]
- Пользователь должен иметь возможность задать/изменить описание ноды в панели свойств.

### FR-3.5. Удалить node [P0]
- При удалении ноды должны удаляться все связанные с ней edge.
- Ссылки drill-down, привязанные к удаляемой ноде, должны быть удалены.

---

## FR-4. Операции над связями (edges)

### FR-4.1. Создать edge [P0]
- Пользователь должен иметь возможность создать направленную стрелку между двумя нодами.
- Edge хранится по `sourceId` и `targetId`.
- В MVP edge не имеет типа — только направление.

### FR-4.2. Изменить подпись edge [P1]
- Пользователь должен иметь возможность задать текстовую подпись для edge.

### FR-4.3. Удалить edge [P0]
- Пользователь должен иметь возможность удалить связь.

---

## FR-5. Операции над frame

### FR-5.1. Создать frame [P0]
- Пользователь должен иметь возможность создать frame — контейнер для группировки нод.
- Frame автоматически получает стабильный `id`.

### FR-5.2. Задать имя и описание frame [P0]
- Пользователь должен иметь возможность задать `name` и `description` для frame.

### FR-5.3. Изменить состав frame [P0]
- Пользователь должен иметь возможность добавлять ноды во frame и убирать их из frame.

### FR-5.4. Удалить frame [P1]
- При удалении frame ноды, входящие в него, не удаляются — они остаются на canvas.

---

## FR-6. Операции drill-down

### FR-6.1. Добавить drill-down к node [P0]
- В панели свойств ноды пользователь должен видеть секцию drill-down.
- Пользователь должен иметь возможность: создать новую дочернюю модель или привязать существующую.

### FR-6.2. Перейти в drill-down модель [P0]
- Пользователь должен иметь возможность открыть дочернюю модель из панели свойств ноды.
- Canvas должен отобразить содержимое дочерней модели.

### FR-6.3. Вернуться из drill-down [P0]
- Система должна предоставить навигацию «назад» (кнопка назад и/или хлебные крошки).
- При возврате canvas показывает родительскую модель.

### FR-6.4. Множественные drill-down [P0]
- Одна нода может ссылаться на несколько дочерних моделей.
- Пользователь должен видеть список всех drill-down и выбирать нужный.

### FR-6.5. Удалить drill-down ссылку [P1]
- Пользователь должен иметь возможность отвязать дочернюю модель от ноды.
- Сама модель при этом не удаляется — удаляется только ссылка.

---

## FR-7. Операции step-up

### FR-7.1. Выполнить step-up для frame [P0]
- Пользователь должен иметь возможность запустить step-up для frame.
- Система должна автоматически:
  - Создать новую верхнеуровневую модель.
  - Создать в ней ноду, представляющую данный frame.
  - Установить связь между frame и верхнеуровневой нодой.

### FR-7.2. Перейти в upper-level модель [P0]
- Пользователь должен иметь возможность перейти к модели, созданной через step-up.

### FR-7.3. Вернуться из upper-level [P0]
- Система должна поддерживать обратный переход из upper-level модели к исходной.

---

## FR-8. Операции поздней типизации

### FR-8.1. Назначить тип node [P0]
- Пользователь должен иметь возможность назначить тип любой ноде через панель свойств.
- При назначении типа нода должна визуально измениться (цвет).

### FR-8.2. Изменить тип node [P0]
- Пользователь должен иметь возможность изменить ранее назначенный тип.

### FR-8.3. Снять тип с node [P0]
- Пользователь должен иметь возможность вернуть ноду в нетипизированное состояние.

---

## FR-9. Операции с нотациями

### FR-9.1. Создать нотацию из текущей модели [P0]
- Система должна извлечь список уникальных типов и их цветов из текущей модели.
- Система должна создать новый YAML-файл нотации в каталоге `notations/`.
- Текущая модель должна быть автоматически связана с созданной нотацией.
- Исходная диаграмма не должна теряться или пересоздаваться.

### FR-9.2. Создать модель по существующей нотации [P0]
- При создании новой модели пользователь должен иметь возможность выбрать нотацию.
- Типы из нотации должны быть доступны в контекстном меню canvas.

---

## FR-10. Навигация и контекст

### FR-10.1. Навигация через дерево проекта [P0]
- Пользователь должен иметь возможность переключаться между моделями через левую панель.

### FR-10.2. Хлебные крошки [P0]
- Система должна отображать текущий путь навигации (хлебные крошки), чтобы пользователь понимал, на каком уровне он находится.

### FR-10.3. Кнопка «назад» [P0]
- Система должна поддерживать возврат к предыдущей модели при навигации через drill-down и step-up.

### FR-10.4. Контекст текущей модели [P1]
- Если объект не выбран, панель свойств справа должна показывать свойства текущей модели (имя, нотация, путь).

---

## FR-11. Сохранение и устойчивость

### FR-11.1. Сохранение проекта [P0]
- Все изменения в моделях должны сохраняться в YAML-файлы.
- Периодическое автосохранение в фоне.

### FR-11.2. Повторное открытие [P0]
- Проект должен открываться после закрытия и перезапуска без потери данных: все модели, связи, drill-down, step-up, типы и нотации сохранены.

---

## Обязательный минимальный набор для MVP

Следующие 13 операций являются критичными для MVP (все P0):

1. Создать проект.
2. Открыть проект.
3. Создать freeform-модель.
4. Создать node.
5. Создать edge.
6. Создать frame.
7. Выполнить step-up.
8. Создать drill-down.
9. Перейти между связанными моделями (drill-down / step-up / назад).
10. Назначить тип node.
11. Создать нотацию из модели.
12. Создать typed-модель по нотации.
13. Сохранить и повторно открыть проект.
