# 07. Domain Model

## Обзор сущностей

MVP оперирует 10 основными сущностями. Каждая сущность имеет стабильный `id`, не зависящий от `label` или имени.

```
Project ──> Manifest
   │
   ├──> Model (freeform | typed)
   │      ├──> Node
   │      │      ├──> Drill-down Link ──> Model
   │      │      └──> type (optional) ──> Notation.ObjectType
   │      ├──> Edge
   │      ├──> Frame
   │      │      └──> Step-up Link ──> Model + Node
   │      └──> notation reference ──> Notation
   │
   └──> Notation
          └──> ObjectType[]
```

---

## 1. Project

**Смысл**: верхний уровень организации. Проект — это папка с файлами моделей, нотаций и manifest-файлом.

**Роль**: объединяет модели, задаёт контекст, является верхним уровнем навигации.

**Свойства**:

| Поле | Тип | Описание |
|------|-----|----------|
| id | string | Уникальный идентификатор проекта |
| name | string | Отображаемое имя проекта |

**Правила**:
- Проект идентифицируется по id, а не по имени папки.
- У проекта нет обязательной одной главной модели — на верхнем уровне может быть несколько независимых моделей.

---

## 2. Project Manifest

**Смысл**: служебный файл проекта (`project.yaml`).

**Свойства**:

| Поле | Тип | Описание |
|------|-----|----------|
| id | string | id проекта |
| name | string | Имя проекта |
| defaultModel | string (path) | Путь к стартовой модели (опционально) |
| notations | string[] (paths) | Список путей к файлам нотаций |

**Правила**:
- Manifest — точка входа при открытии проекта.
- Список нотаций может быть пустым.
- Поле `defaultModel` опционально.

---

## 3. Notation

**Смысл**: YAML-файл, определяющий набор типов объектов для использования в моделях.

**Свойства**:

| Поле | Тип | Описание |
|------|-----|----------|
| id | string | Уникальный идентификатор нотации |
| name | string | Отображаемое имя нотации |
| types | ObjectType[] | Список типов объектов |
| recommendedDrilldowns | map (опционально) | Рекомендуемые нотации для детализации по типам |

**ObjectType**:

| Поле | Тип | Описание |
|------|-----|----------|
| id | string | Идентификатор типа |
| name | string | Отображаемое имя типа |
| color | string (hex) | Цвет объектов данного типа |

**Правила**:
- Нотация не редактируется в UI — создаётся автоматически из модели или правится вручную в VS Code.
- `recommendedDrilldowns` — необязательное поле, задел на будущее.

---

## 4. Model

**Смысл**: YAML-файл, хранящий диаграмму — центральная рабочая единица продукта.

**Свойства**:

| Поле | Тип | Описание |
|------|-----|----------|
| id | string | Уникальный идентификатор модели |
| name | string | Отображаемое имя |
| notation | string | `freeform` или id нотации (совпадает с полем `id` в YAML-файле нотации; разрешение id → файл происходит через manifest) |
| nodes | Node[] | Список нод |
| edges | Edge[] | Список стрелок |
| frames | Frame[] | Список frame |

**Правила**:
- Модель может быть freeform (`notation: freeform`) или typed (ссылка на id нотации).
- Freeform — полноценный режим, не временный черновик.
- Модель становится дочерней, когда на неё ссылается нода через drill-down или она создана через step-up.
- Модель идентифицируется по id, переименование не меняет id.

---

## 5. Node

**Смысл**: базовый объект диаграммы.

**Свойства**:

| Поле | Тип | Описание |
|------|-----|----------|
| id | string | Стабильный id внутри модели |
| label | string | Отображаемое имя |
| description | string | Описание (опционально) |
| type | string \| null | id типа из нотации или null |
| position | {x, y} | Координаты на canvas |
| drilldowns | string[] (paths) | Пути к детализирующим моделям |

**Правила**:
- Переименование label не меняет id.
- Все ссылки строятся по принципу `relativeModelPath + nodeId`.
- В freeform-модели type может быть null.
- Одна нода может иметь несколько drill-down.

---

## 6. Edge

**Смысл**: направленная стрелка между двумя нодами.

**Свойства**:

| Поле | Тип | Описание |
|------|-----|----------|
| id | string | Идентификатор связи |
| source | string | id ноды-источника |
| target | string | id ноды-приёмника |
| label | string (опционально) | Текстовая подпись |

**Правила**:
- В MVP нет типизации edges — только направление.
- При удалении ноды все связанные edges удаляются автоматически.

---

## 7. Frame

**Смысл**: контейнер, выделяющий группу объектов и выражающий их принадлежность к более верхнему объекту. Frame — это не просто декоративная рамка, а носитель смысловой связи с операцией step-up.

**Свойства**:

| Поле | Тип | Описание |
|------|-----|----------|
| id | string | Стабильный id внутри модели |
| name | string | Имя frame |
| description | string (опционально) | Описание |
| nodes | string[] | Список id нод, входящих во frame |
| stepUp | StepUpLink \| null | Ссылка на верхнеуровневую модель и ноду |

**Правила**:
- При удалении frame ноды не удаляются, остаются на canvas.
- Frame — промежуточная форма между группировкой и абстрагированием.

---

## 8. Drill-down Link

**Смысл**: ссылка от ноды к дочерней модели, позволяющая раскрыть объект в более детальное представление.

**Структура**: хранится как элемент массива `drilldowns` в Node.

**Адресация**: относительный путь к YAML-файлу модели.

**Правила**:
- Одна нода может иметь несколько drill-down.
- Удаление ссылки не удаляет саму модель.
- Если ссылка указывает на несуществующий файл, система показывает сообщение и предлагает создать модель.

---

## 9. Step-up Link

**Смысл**: ссылка от frame к верхнеуровневой модели и ноде в ней, позволяющая подняться на уровень абстракции.

**Структура**: хранится как объект `stepUp` в Frame.

**Свойства**:

| Поле | Тип | Описание |
|------|-----|----------|
| model | string (path) | Путь к верхнеуровневой модели |
| nodeId | string | id ноды в верхнеуровневой модели, представляющей данный frame |

---

## 10. Model Reference

**Смысл**: любая ссылка между моделями (drill-down, step-up, notation reference).

**Адресация**: все ссылки используют относительные пути к YAML-файлам моделей.

**Правила**:
- Адресация объектов: `modelPath + nodeId` или `modelPath + frameId`.
- Ссылки по идентификаторам, а не по именам.

---

## Отношения между сущностями

| Связь | Тип | Описание |
|-------|-----|----------|
| Project → Model | 1:N | Проект содержит несколько моделей |
| Project → Notation | 1:N | Проект содержит несколько нотаций |
| Model → Node | 1:N | Модель содержит ноды |
| Model → Edge | 1:N | Модель содержит стрелки |
| Model → Frame | 1:N | Модель содержит frame |
| Model → Notation | N:1 | Typed-модель ссылается на одну нотацию |
| Node → Model (drill-down) | 1:N | Нода может ссылаться на несколько дочерних моделей |
| Frame → Model+Node (step-up) | 1:1 | Frame может быть связан с одной верхнеуровневой нодой |
| Notation → ObjectType | 1:N | Нотация определяет набор типов |

## Особенности предметной модели MVP

- Freeform и typed-модели хранятся в одном формате. Различие — только значение поля `notation`.
- Нотация — это не метамодель в полном смысле, а минимальный набор типов с цветами.
- Frame — не обычный typed-объект, а особый контейнер со смысловой связью step-up.
- В MVP нет сквозных инстансов объектов между моделями — каждая модель содержит свои ноды.
