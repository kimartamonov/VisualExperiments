# 12. Risks, Decisions, Open Questions

## Принятые решения (Architecture Decision Records)

### ADR-1. Canvas: React Flow

- **Решение**: использовать React Flow как основу canvas.
- **Статус**: подтверждено прототипированием.
- **Обоснование**: поддерживает frame, drag-and-drop, edges. Наиболее подходящий вариант на данный момент.
- **Ограничения**: вложенность в нужном виде напрямую не подтверждена, но drill-down реализуется как ссылка в свойствах ноды, а не как встроенная функция React Flow.
- **«Красная линия»**: React Flow предпочтительно не менять.

### ADR-2. Формат хранения: YAML

- **Решение**: YAML как основной формат для моделей, нотаций, manifest.
- **Статус**: принято, но не является абсолютным ограничением.
- **Обоснование**: читаемость для человека, возможность редактирования в VS Code.
- **Готовность к изменению**: при необходимости допустим переход на JSON или SQLite, если это улучшит архитектуру без потери читаемости.

### ADR-3. Backend: файловое хранение без БД

- **Решение**: данные хранятся в файловой системе. База данных не используется.
- **Статус**: принято для MVP.
- **Обоснование**: простота, прозрачность структуры, возможность копировать проекты как папки.
- **Готовность к изменению**: если другой вариант улучшит архитектуру или производительность — допустим переход.

### ADR-4. TypeScript

- **Решение**: использовать TypeScript для frontend и backend.
- **Статус**: предпочтителен, но не является принципиальным требованием.

### ADR-5. Однопользовательский режим

- **Решение**: MVP не поддерживает совместную работу.
- **Статус**: зафиксировано для MVP.
- **Обоснование**: фокус на проверке гипотезы, а не на multi-user функциональности.

### ADR-6. Нотации не редактируются в UI

- **Решение**: нотации создаются автоматически из модели или правятся вручную в VS Code.
- **Статус**: зафиксировано для MVP.
- **Обоснование**: визуальный редактор нотаций — значительная дополнительная сложность.

### ADR-7. Drill-down как ссылка, а не вложенность React Flow

- **Решение**: drill-down реализуется через ссылки в свойствах ноды, а не через встроенный механизм вложенности React Flow.
- **Статус**: принято.
- **Обоснование**: прототипирование не подтвердило вложенность React Flow в нужном виде. Ссылочный подход проще и достаточен.

### ADR-8. Принятие решений

- **Решение**: бизнес-компромиссы принимает заказчик. Технические архитектурные решения — разработчик, в рамках бизнес-ограничений.
- **Статус**: зафиксировано.

## Оценка рисков

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|-------------|---------|-----------|
| R1 | React Flow не поддерживает нужное поведение step-up (автоматическое создание верхнеуровневой модели) | Средняя | Высокое | Step-up — это логическая операция Model Semantics Layer, а не функция React Flow. Кастомная реализация возможна |
| R2 | YAML неудобен для больших моделей (>100 нод) | Низкая (MVP: до 50 нод) | Среднее | Допустим переход на JSON/SQLite. Для MVP объёмы малы |
| R3 | Автосохранение создаёт фризы на canvas | Средняя | Среднее | Debounce, фоновое сохранение, оптимизация частоты |
| R4 | Навигация между уровнями (drill-down, step-up) неинтуитивна для пользователя | Средняя | Высокое | Хлебные крошки + кнопка назад. Итеративное улучшение по результатам использования |
| R5 | Потеря данных при сбое сервера | Средняя | Высокое | Автосохранение. Данные на диске. При перезапуске — восстановление из последнего сохранения |
| R6 | Конфликт данных при ручном редактировании YAML параллельно с работой в UI | Низкая (не целевой сценарий) | Среднее | Документировано как ограничение. Last-write-wins |
| R7 | Сложность реализации «Create notation from model» (извлечение типов, связывание) | Средняя | Среднее | Начать с минимальной реализации: только уникальные типы + цвета |

## Главный открытый вопрос

> Насколько естественно для пользователя действительно выращивать нотацию из свободной практики моделирования, а затем использовать её как устойчивый рабочий язык?

Этот вопрос является основной проверяемой гипотезой MVP и не может быть разрешён до получения реального опыта использования.

## Следующая волна после MVP

По результатам использования MVP, предполагаемые направления развития:

1. **UX переходов между уровнями** — улучшение навигации drill-down / step-up.
2. **Развитие нотаций** — richer types, типизированные связи, визуальное редактирование.
3. **Логическая структура проекта** — развитие связи между файловой и смысловой иерархией.
4. **Повторное использование** — сквозные объекты, библиотеки нотаций.
5. **Интеграции** — Windmill, Git, синхронизация с внешними папками.
