# 10. Roadmap and Delivery Plan

## Общий контекст

- Жёсткого календарного дедлайна нет.
- Внешние факторы, влияющие на сроки, не определены.
- Разработка ведётся последовательно, в значительной степени с использованием кодовых агентов.
- Состав команды и доступность разработчиков не фиксируются как ограничение.
- Промежуточные milestone не являются обязательным требованием.

---

## Этапы реализации

Рекомендуемая последовательность реализации MVP, основанная на зависимостях между функциями.

### Этап 1. Фундамент: проект и файловая структура

**Цель**: рабочий backend и базовый UI для управления проектами.

**Содержание**:
- Backend: Node.js сервер, REST API для проектов.
- Файловые операции: создание/чтение/запись YAML.
- Frontend: экран списка проектов, создание/открытие проекта.
- Левая панель: отображение файловой структуры проекта.
- Manifest v1 и file placement contract: `project.yaml` с минимальными полями, `models/` при bootstrap и `notations/` по требованию.

**Результат**: можно создать проект, увидеть его структуру, backend умеет работать с файлами, а правила путей для create model, drill-down и step-up не требуют дополнительных product-решений.

### Этап 2. Canvas и базовое моделирование

**Цель**: рабочий визуальный редактор для freeform-диаграмм.

**Содержание**:
- React Flow canvas в центральной панели.
- Создание, перемещение, удаление нод.
- Создание и удаление направленных edges.
- Правая панель свойств (label, description).
- Создание и открытие freeform-модели.
- Сохранение модели в YAML.

**Результат**: пользователь может создать freeform-модель, нарисовать диаграмму, сохранить и заново открыть.

### Этап 3. Frame, drill-down, step-up

**Цель**: поддержка уровней абстракции.

**Содержание**:
- Создание frame, добавление/удаление нод из frame.
- Drill-down: добавление ссылки, создание/привязка дочерней модели, переход и возврат.
- Step-up: автоматическое создание верхнеуровневой модели, переход и возврат.
- Навигация: хлебные крошки, кнопка «назад».

**Результат**: работают переходы между уровнями в обе стороны.

### Этап 4. Типизация и нотации

**Цель**: поздняя типизация и создание нотаций из модели.

**Содержание**:
- Назначение типов нодам в панели свойств.
- Визуальное отражение типа (цвет).
- Создание нотации из текущей модели (извлечение типов).
- Создание typed-модели по существующей нотации.
- Типы из нотации доступны в контекстном меню canvas.

**Результат**: полный цикл freeform → типизация → нотация → typed-модель работает.

### Этап 5. Стабилизация и автосохранение

**Цель**: устойчивая работа всего MVP.

**Содержание**:
- Автосохранение в фоне.
- Обработка ошибок: невалидный YAML, битые ссылки.
- Проверка сквозного демо-сценария целиком.
- Исправление найденных проблем.

**Результат**: MVP проходит полный демо-сценарий из 14 шагов без сбоев (см. 04_User_Scenarios, раздел «Сквозной демонстрационный сценарий MVP»).

---

## Зависимости между этапами

```
Этап 1 (Фундамент)
   │
   ▼
Этап 2 (Canvas)
   │
   ▼
Этап 3 (Frame, drill-down, step-up)
   │
   ▼
Этап 4 (Типизация и нотации)
   │
   ▼
Этап 5 (Стабилизация)
```

Этапы строго последовательны: каждый следующий зависит от результатов предыдущего.

---

## Что нужно для старта разработки

- Зафиксированные требования (данный пакет документов).
- Настроенное окружение: Node.js, React, React Flow.
- Репозиторий проекта.

## Что нужно для пилотного использования

- Работающий MVP, проходящий сквозной демо-сценарий.
- Возможность создать реальный проект (например, Business Model Canvas) и проверить drill-down.

## Что нужно для дальнейшего развития

- Результаты использования MVP заказчиком.
- Решения по открытым вопросам (типизированные связи, визуальный редактор нотаций, интеграция с Windmill).

---

## Риски по срокам

| Риск | Влияние | Митигация |
|------|---------|-----------|
| React Flow не поддерживает нужное поведение для step-up | Задержка этапа 3 | Прототипирование проведено; при проблемах — кастомная реализация |
| Сложность навигации между уровнями | Задержка этапа 3 | Начать с простой реализации (кнопка назад), усложнять итеративно |
| YAML неудобен для больших моделей | Задержка этапа 5 | Допустим переход на JSON/SQLite при сохранении читаемости |
| Автосохранение влияет на производительность canvas | Задержка этапа 5 | Debounce, оптимизация частоты сохранений |

---

## Приоритизация при нехватке ресурсов

Если потребуется сократить scope, рекомендуемый порядок отказа (от наименее критичного):

1. Автосохранение (заменить на ручное сохранение — автосохранение является P1).
2. Step-up (отложить, сохранив frame как визуальный элемент).
3. Множественные drill-down (упростить до одного drill-down на ноду).
4. Поздняя типизация и создание нотации (критичны, но могут быть упрощены).
5. Freeform-моделирование + базовый canvas (не сокращается — это ядро).
