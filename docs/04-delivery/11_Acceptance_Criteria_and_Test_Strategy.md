# 11. Acceptance Criteria and Test Strategy

## Процесс приёмки

- Результат принимает **заказчик лично** через ручную проверку ключевых сценариев.
- Автоматизированные тесты в MVP не требуются (вопрос может быть пересмотрен позже).
- Мелкие визуальные дефекты **не блокируют** приёмку, если основная логика работает.

---

## Definition of Done (для каждой задачи)

Задача считается выполненной, когда:

1. Функция работает в соответствии с описанием в функциональных требованиях.
2. Данные корректно сохраняются в YAML и восстанавливаются при повторном открытии.
3. Функция не ломает работу ранее реализованных функций.
4. UI не содержит критических дефектов (аварийное завершение, потеря данных, невозможность выполнить действие).

---

## Критерии приёмки по функциям

### AC-1. Создание проекта
- При создании проекта появляется папка с `project.yaml` и каталогом `models/`.
- Проект отображается в списке проектов.
- Проект можно открыть и увидеть пустую структуру.

### AC-2. Файловая структура
- Левая панель показывает реальную файловую структуру проекта (папки и YAML-файлы).
- При создании новой модели она появляется в дереве.

### AC-3. Freeform-модель
- Можно создать модель без выбора нотации.
- Открыт пустой canvas.
- В YAML-файле `notation: freeform`.

### AC-4. Создание и редактирование нод
- Ноды создаются через контекстное меню canvas.
- Ноды перемещаются drag-and-drop.
- Label и description редактируются в панели свойств.
- При удалении ноды связанные edges удаляются.

### AC-5. Создание edges
- Направленная стрелка создаётся между двумя нодами.
- Edge отображается визуально с направлением.
- Edge можно удалить.

### AC-6. Frame
- Frame создаётся, отображается как визуальный контейнер.
- Ноды можно добавлять и убирать из frame.
- Frame имеет name и description.
- При удалении frame ноды остаются на canvas.

### AC-7. Drill-down
- В панели свойств ноды есть секция drill-down.
- Можно создать новую дочернюю модель или привязать существующую.
- Переход в дочернюю модель открывает её на canvas.
- Возврат работает через кнопку «назад» и/или хлебные крошки.
- Если файл дочерней модели не найден — показывается сообщение «Модель не найдена. Создать?».

### AC-8. Step-up
- Для frame можно запустить step-up.
- Автоматически создаётся верхнеуровневая модель с нодой, представляющей frame.
- Переход в верхнеуровневую модель и обратно работает.

### AC-9. Поздняя типизация
- В freeform-модели можно назначить тип любой ноде.
- Нода визуально меняется (цвет) после назначения типа.
- Тип можно изменить или снять.
- Назначение типов не разрушает существующую модель (связи, позиции сохраняются).

### AC-10. Создание нотации из модели
- Из модели с типизированными нодами можно создать нотацию.
- Система извлекает уникальные типы и цвета.
- Создаётся YAML-файл нотации в `notations/`.
- Текущая модель связывается с нотацией.
- Исходная диаграмма не теряется.

### AC-11. Typed-модель по нотации
- Можно создать новую модель, выбрав существующую нотацию.
- Типы из нотации доступны в контекстном меню canvas.
- Создаваемые ноды получают тип и цвет из нотации.

### AC-12. Сохранение и повторное открытие
- Проект сохраняется (автосохранение или ручное).
- После закрытия и повторного открытия: все модели, ноды, edges, frames, drill-down, step-up, типы, нотации — сохранены.
- Структура проекта не нарушена.

### AC-13. Обработка ошибок
- Невалидный YAML: сообщение об ошибке, модель не загружается, приложение не падает.
- Битая drill-down ссылка: сообщение «Модель не найдена. Создать?».

## Сквозной демонстрационный сценарий

Основной критерий приёмки MVP — успешное прохождение полного демо-сценария из 14 шагов (см. документ 04_User_Scenarios, раздел «Сквозной демонстрационный сценарий MVP»).

### Критерий «сломанного MVP»

MVP считается **неприемлемым**, если невозможно пройти основную логику сценария целиком. Отдельные мелкие недоработки допустимы, если базовый сценарий работы системы в целом выполняется.

## Тестовые данные

### Рекомендуемый тестовый use case

Business Model Canvas — создать как freeform-диаграмму, назначить типы элементам, извлечь нотацию, затем выполнить drill-down из одного элемента в более детальную структуру.

### Ориентиры для тестирования

- Проект с 5–10 моделями.
- Диаграмма с 20–50 нодами.
- 1–2 уровня drill-down.
- Минимум 1 step-up.
- Минимум 1 нотация, созданная из модели.
- Минимум 1 typed-модель, созданная по нотации.

---

## Стратегия тестирования

### MVP

| Тип | Метод | Ответственный |
|-----|-------|--------------|
| Функциональное тестирование | Ручная проверка по критериям приёмки (AC-1 — AC-13) | Заказчик |
| Сквозное тестирование | Прохождение демо-сценария целиком | Заказчик |
| Smoke-тестирование | Проверка базовых операций после каждого этапа | Разработчик |

### После MVP (к обсуждению)

- Unit-тесты для Model Semantics Layer.
- Integration-тесты для REST API.
- E2E-тесты для сквозного сценария.

## Приоритизация дефектов

| Приоритет | Описание | Блокирует приёмку? |
|-----------|----------|--------------------|
| **Критический** | Невозможно выполнить шаг демо-сценария. Потеря данных. Аварийное завершение. | Да |
| **Высокий** | Функция работает, но с существенным неудобством (например, нет обратной навигации). | Да, если затрагивает ключевые операции |
| **Средний** | Мелкий визуальный дефект. Незначительное отклонение от ожидаемого поведения. | Нет |
| **Низкий** | Косметические недоработки. Улучшения UX. | Нет |
