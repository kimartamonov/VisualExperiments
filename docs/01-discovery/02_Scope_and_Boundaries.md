# 02. Scope and Boundaries

## Что входит в MVP

### Работа с проектами
- Список проектов.
- Создание нового пустого проекта (папка + `project.yaml` + каталог `models/`).
- Вход в существующий проект.
- Просмотр файловой структуры проекта (папки и YAML-файлы).
- Создание папок внутри проекта.

### Работа с моделями
- Создание freeform-модели (без нотации).
- Создание typed-модели (по существующей нотации).
- Открытие, редактирование, переименование модели.
- Сохранение модели в YAML.
- Повторное открытие модели без потери структуры.

### Визуальный редактор (canvas)
- React Flow как основа canvas.
- Создание, перемещение, удаление нод (node).
- Направленные стрелки (edge) без типизации.
- Frame — контейнер для группировки объектов.
- Панель свойств выбранного объекта справа.
- Внешний вид объектов в стиле FigJam: прямоугольники со скруглёнными углами, цветовая дифференциация.

### Операции уровней абстракции
- **Drill-down**: раскрытие ноды в одну или несколько дочерних моделей.
- **Step-up**: сборка frame в верхнеуровневый объект с автоматическим созданием верхнеуровневой модели.
- Навигация между уровнями: кнопка «назад», хлебные крошки.

### Типизация и нотации
- Поздняя типизация: назначение типов объектам после создания диаграммы.
- Создание нотации из существующей модели (извлечение типов и цветов).
- YAML-нотации, читаемые с диска.
- Создание новой модели по существующей нотации.

### Хранение
- Файлово-ориентированное хранение на сервере.
- YAML как основной формат моделей, нотаций и manifest.
- Ручное сохранение. Автосохранение в фоне желательно, но не обязательно для MVP.

## Что сознательно не входит в MVP

| Исключение | Причина |
|---|---|
| Типизированные связи (edges) | Усложняет MVP без пользы для проверки гипотезы |
| Визуальный редактор метамоделей / нотаций | Нотации создаются автоматически из модели или правятся вручную в VS Code |
| Формальная валидация модели | MVP — исследовательская песочница, не production-инструмент |
| Совместная работа нескольких пользователей | Однопользовательский режим |
| История изменений / версионирование | Не приоритет для проверки гипотезы |
| Роли и права доступа | Не требуется для одного пользователя |
| Компиляция / генерация в Windmill | Задел на будущее, не MVP |
| Сквозные инстансы объектов между диаграммами | Сложная функция вне MVP |
| Сложные shape / иконки / rich styling | Минимальная визуальная вариативность через цвет |
| Plugin system | Преждевременная абстракция |
| Real-time collaboration | Не нужно для одного пользователя |
| База данных | Файловое хранение достаточно для MVP |

## Границы MVP

- **Размер диаграммы**: до 50 объектов (допустимо до 100 в отдельных случаях). Сложность снимается через декомпозицию и drill-down.
- **Размер проекта**: 5–10 моделей в одном проекте.
- **Глубина drill-down**: 1–2 уровня.
- **Платформа**: десктоп, современные браузеры. Мобильные устройства и планшеты не поддерживаются.
- **Аутентификация**: не требуется. Допустима простая заглушка с одним паролем для работы в доверенной сети.

## Предположения

- Пользователь понимает смысл диаграмм, типов объектов и уровней детализации.
- Пользователь готов при необходимости редактировать YAML вручную во внешнем редакторе.
- Проект представляет собой папку, которую можно копировать и переносить вручную.
- Одновременное редактирование одного файла в VS Code и визуальном редакторе не является целевым сценарием.

## Зависимости

- React Flow — основная библиотека для canvas. Прототипирование подтвердило поддержку frame. Drill-down реализуется не через встроенный механизм React Flow, а через ссылки в свойствах ноды.
- Node.js — серверная среда для файлового backend.
- YAML — формат хранения (при необходимости может быть заменён на JSON или SQLite без принципиальных ограничений).

## Минимальный критерий полноты MVP

Пользователь способен пройти сквозной демонстрационный сценарий:
создание проекта → freeform-моделирование → frame → step-up → drill-down → поздняя типизация → создание нотации → создание typed-модели → сохранение и повторное открытие без потери структуры.
