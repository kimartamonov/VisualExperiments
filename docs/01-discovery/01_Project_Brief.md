# 01. Project Brief

## Название проекта

VisualExperiments — визуальный редактор моделей и нотаций.

## Краткая идея

Веб-приложение для визуального моделирования, в котором пользователь может работать как со свободными диаграммами, так и с диаграммами по собственным YAML-нотациям. Ключевая особенность — поддержка двух направлений: **вниз по детализации** (drill-down) и **вверх по абстракции** (step-up), а также возможность **извлекать новую нотацию из уже нарисованной свободной диаграммы**.

## Бизнес-проблема / пользовательская проблема

Существующие инструменты моделирования (BPMN-редакторы, enterprise-модеры, FigJam/Miro) работают в одном из двух режимов: либо требуют заранее определённую нотацию, либо предоставляют полностью свободный canvas без семантики. Ни один из них не поддерживает естественный путь мышления, при котором пользователь:

1. Сначала рисует свободную карту.
2. Постепенно различает в ней устойчивые типы.
3. Фиксирует типы как нотацию.
4. Продолжает моделирование уже в рамках этой нотации.

Кроме того, большинство инструментов не поддерживают одновременное движение по уровням абстракции в обе стороны.

## Цель проекта

Создать минимально рабочую исследовательскую среду (MVP), в которой можно проверить ключевую гипотезу: удобно ли и полезно ли пользователю выращивать нотацию из свободной практики моделирования, а затем использовать её как устойчивый рабочий язык.

## Ожидаемый результат

Работающее веб-приложение, позволяющее пройти полный цикл:

**свободное моделирование → структурирование → выделение типов → фиксация нотации → повторное моделирование по нотации**

с поддержкой drill-down и step-up переходов между уровнями.

## Заказчик

Автор продуктовой идеи — единственный заказчик и основной пользователь первой версии. Принимает решения по бизнес-компромиссам. Технические архитектурные решения делегирует разработчику на основе зафиксированных бизнес-ограничений.

## Основные заинтересованные стороны

- **Заказчик / основной пользователь** — определяет приоритеты, принимает результат.
- **Технический специалист / разработчик** — принимает архитектурные решения в рамках бизнес-ограничений.

## Что считается успехом

MVP считается успешным, если пользователь может без внешних обходных путей пройти демонстрационный сценарий из 14 шагов (см. 04_User_Scenarios), включая:

1. Создание проекта и freeform-модели.
2. Построение диаграммы с объектами, стрелками, frame.
3. Выполнение step-up и drill-down.
4. Позднюю типизацию объектов.
5. Создание нотации из модели.
6. Создание новой typed-модели по нотации.
7. Сохранение и повторное открытие без потери структуры.

## Известные ограничения

- Однопользовательский режим (нет совместной работы).
- Нет типизированных связей.
- Нет визуального редактора метамоделей/нотаций.
- Нет валидации моделей.
- Нет версионирования и истории изменений.
- Нет интеграции с Windmill или другими исполняющими системами.
- Нет сложных визуальных форм — базовые прямоугольники со скруглёнными углами.

## Что пока неясно

- Финальная YAML-схема моделей и нотаций (будет уточняться экспериментально).
- Точное соотношение файловой и логической структуры моделей.
- Семантика frame в более зрелой версии.
- Способ отображения режимов freeform и typed в UI.
