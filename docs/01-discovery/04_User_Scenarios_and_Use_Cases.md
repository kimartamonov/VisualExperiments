# 04. User Scenarios and Use Cases

## Ключевые пользовательские сценарии

### Сценарий 1. Вход и выбор проекта

- **Актор**: пользователь.
- **Цель**: войти в рабочее пространство.
- **Предусловия**: приложение запущено.
- **Основной сценарий**: пользователь видит список проектов, выбирает существующий или создаёт новый.
- **Результат**: пользователь находится внутри проекта, видит структуру файлов.

### Сценарий 2. Создание проекта

- **Актор**: пользователь.
- **Цель**: создать новое рабочее пространство.
- **Предусловия**: пользователь на экране списка проектов.
- **Основной сценарий**: пользователь нажимает «Создать проект», вводит имя.
- **Результат**: создана папка проекта с `project.yaml` и каталогом `models/`. Пользователь попадает в рабочее пространство проекта.

### Сценарий 3. Просмотр структуры проекта

- **Актор**: пользователь.
- **Цель**: понять состав проекта.
- **Предусловия**: пользователь внутри проекта.
- **Основной сценарий**: слева отображается дерево папок и YAML-файлов моделей.
- **Результат**: пользователь видит файловую структуру проекта.

### Сценарий 4. Создание freeform-модели

- **Актор**: пользователь.
- **Цель**: начать свободное моделирование.
- **Предусловия**: пользователь внутри проекта.
- **Основной сценарий**: пользователь создаёт новую модель без выбора нотации.
- **Результат**: открыт пустой canvas для свободной диаграммы. Модель сохранена как YAML с `notation: freeform`.

### Сценарий 5. Построение свободной диаграммы

- **Актор**: пользователь.
- **Цель**: визуализировать идею.
- **Предусловия**: открыта freeform-модель.
- **Основной сценарий**:
  1. Правый клик по canvas → «Create new» → выбор свободного объекта.
  2. Создание нескольких нод.
  3. Перемещение нод на canvas.
  4. Соединение нод направленными стрелками.
- **Результат**: на canvas присутствует диаграмма из объектов и связей.

### Сценарий 6. Редактирование свойств объекта

- **Актор**: пользователь.
- **Цель**: уточнить параметры объекта.
- **Предусловия**: на canvas есть ноды.
- **Основной сценарий**: пользователь выбирает ноду, справа открывается панель свойств. Пользователь изменяет label, description, тип (если доступен).
- **Результат**: свойства ноды обновлены.

### Сценарий 7. Создание frame

- **Актор**: пользователь.
- **Цель**: сгруппировать объекты для последующего step-up.
- **Предусловия**: на canvas есть несколько нод.
- **Основной сценарий**: пользователь создаёт frame, задаёт ему имя, перетаскивает ноды внутрь.
- **Результат**: ноды визуально объединены во frame с заданным именем.

### Сценарий 8. Step-up

- **Актор**: пользователь.
- **Цель**: поднять группу элементов на верхний уровень абстракции.
- **Предусловия**: на canvas существует frame с нодами.
- **Основной сценарий**:
  1. Пользователь выбирает frame.
  2. Запускает операцию step-up.
  3. Система автоматически создаёт новую верхнеуровневую модель.
  4. В новой модели появляется нода, представляющая данный frame.
  5. Между frame и верхнеуровневой нодой устанавливается связь.
- **Результат**: создана верхнеуровневая модель с нодой, ссылающейся на исходный frame.

### Сценарий 9. Обратный переход из step-up

- **Актор**: пользователь.
- **Цель**: вернуться к детальной модели из верхнего уровня.
- **Предусловия**: пользователь находится в upper-level модели, созданной через step-up.
- **Основной сценарий**: пользователь нажимает кнопку «назад» или использует хлебные крошки.
- **Результат**: пользователь возвращается к исходной модели с frame.

### Сценарий 10. Drill-down

- **Актор**: пользователь.
- **Цель**: раскрыть объект в более детальную модель.
- **Предусловия**: на canvas есть нода.
- **Основной сценарий**:
  1. Пользователь выбирает ноду.
  2. В панели свойств в секции drill-down нажимает «Добавить модель».
  3. Создаёт новую модель или привязывает существующую.
  4. Переходит в drill-down модель.
- **Результат**: открыта дочерняя модель. В свойствах исходной ноды зафиксирована ссылка.

### Сценарий 11. Работа с несколькими детализациями

- **Актор**: пользователь.
- **Цель**: иметь несколько представлений детализации одного объекта.
- **Предусловия**: у ноды уже есть один drill-down.
- **Основной сценарий**: пользователь добавляет ещё одну drill-down модель через панель свойств.
- **Результат**: нода ссылается на несколько дочерних моделей. Пользователь может открыть любую из них.

### Сценарий 12. Поздняя типизация

- **Актор**: пользователь.
- **Цель**: назначить типы объектам после создания диаграммы.
- **Предусловия**: открыта freeform-модель с нодами.
- **Основной сценарий**:
  1. Пользователь выбирает ноду.
  2. В панели свойств назначает тип и цвет.
  3. Повторяет для других нод.
- **Результат**: ноды имеют присвоенные типы и цвета. Модель готова к извлечению нотации.
- **Частота**: центральный сценарий MVP, одна из ключевых продуктовых функций.

### Сценарий 13. Создание нотации из модели

- **Актор**: пользователь.
- **Цель**: зафиксировать найденные типы как reusable-нотацию.
- **Предусловия**: в модели есть ноды с назначенными типами и цветами.
- **Основной сценарий**:
  1. Пользователь запускает операцию «Create notation».
  2. Система извлекает уникальные типы и их цвета из текущей модели.
  3. Создаёт новый YAML-файл нотации в `notations/`.
  4. Текущая модель связывается с созданной нотацией.
- **Результат**: YAML-нотация создана. Текущая модель помечена как typed. Исходная диаграмма не теряется и не пересоздаётся.

### Сценарий 14. Создание typed-модели по нотации

- **Актор**: пользователь.
- **Цель**: создать новую модель в рамках существующей нотации.
- **Предусловия**: в проекте есть хотя бы одна нотация.
- **Основной сценарий**: пользователь создаёт новую модель, выбирая существующую нотацию. В контекстном меню canvas доступны типы из нотации.
- **Результат**: открыта новая модель с доступными типами объектов из нотации.

### Сценарий 15. Сохранение, закрытие и повторное открытие

- **Актор**: пользователь.
- **Цель**: убедиться в сохранности работы.
- **Предусловия**: пользователь работал с проектом.
- **Основной сценарий**: пользователь закрывает проект, затем снова открывает его.
- **Результат**: все модели, связи, drill-down, step-up, типы и нотации сохранены и доступны.

## Сквозной демонстрационный сценарий MVP

Этот сценарий является основным критерием успеха MVP. Все шаги должны работать как единый непрерывный путь.

| Шаг | Действие | Ожидаемый результат | Что проверяется |
|-----|----------|---------------------|-----------------|
| 1 | Создать пустой проект | Появляется проект с `project.yaml` и `models/` | Создание проекта |
| 2 | Войти в рабочее пространство | Видна трёхпанельная структура: дерево, canvas, свойства | Интерфейс редактора |
| 3 | Создать freeform-модель | Открыт пустой canvas, модель в дереве | Freeform как полноценный режим |
| 4 | Нарисовать 4–5 объектов и стрелки | Объекты и связи на canvas | Базовое редактирование |
| 5 | Выделить часть объектов во frame | Ноды визуально сгруппированы | Frame как контейнер |
| 6 | Выполнить step-up | Создана верхнеуровневая модель с нодой для frame | Step-up |
| 7 | Перейти в upper-level модель и вернуться | Навигация работает в обе стороны (кнопка назад / хлебные крошки) | Обратимость переходов |
| 8 | Создать drill-down для одной ноды | Создана дочерняя модель, ссылка в свойствах ноды | Drill-down |
| 9 | Перейти в drill-down и вернуться | Навигация работает | Обратимость drill-down |
| 10 | Назначить типы объектам исходной диаграммы | Ноды получают типы и цвета | Поздняя типизация |
| 11 | Создать нотацию из модели | YAML-файл нотации создан, модель связана с ней | Извлечение нотации |
| 12 | Создать новую модель по этой нотации | Типы из нотации доступны в контекстном меню | Typed-моделирование |
| 13 | Добавить несколько объектов в typed-модели | Объекты создаются с типами из нотации | Работа по нотации |
| 14 | Сохранить и повторно открыть проект | Все данные сохранены, структура не нарушена | Устойчивость состояния |

### Критерий успешности

- Все сущности создаются через UI.
- Переходы между уровнями работают и обратимы.
- Поздняя типизация не разрушает модель.
- Нотация создаётся на основе модели.
- Новая typed-модель создаётся на основе нотации.
- Проект сохраняется и повторно открывается без потери целостности.
