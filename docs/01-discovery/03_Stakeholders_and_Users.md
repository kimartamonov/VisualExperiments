# 03. Stakeholders and Users

## Основные заинтересованные стороны

### Заказчик / владелец продукта

- Единственный заказчик проекта.
- Определяет приоритеты, принимает бизнес-компромиссы.
- Принимает результат через ручную проверку ключевых сценариев.
- Является основным (и единственным на этапе MVP) пользователем продукта.

### Технический специалист / разработчик

- Принимает архитектурные решения в рамках бизнес-ограничений заказчика.
- Реализация ведётся последовательно, с использованием кодовых агентов.

## Основной пользователь

### Профиль

Человек, который самостоятельно разрабатывает фреймворки, схемы рассуждения, верхнеуровневые workflow-карты и экспериментальные нотации. Это не массовый пользователь и не оператор бизнес-процесса, а исследователь, архитектор или автор концептуальных моделей.

### Характер работы

- Работает не от жёстко заданной схемы, а от мысли, наброска, гипотезы.
- Может не знать заранее, какие типы объектов понадобятся.
- Работа носит итеративный и исследовательский характер: набросок → различение → типизация → фиксация нотации → повторное моделирование.
- Может несколько раз возвращаться к одной диаграмме, менять уровни абстракции, переосмыслять типы.

### Уровень подготовки

- Понимает смысл диаграмм, типов объектов, уровней детализации и проектной структуры.
- Не обязан быть разработчиком или специалистом по формальным метамоделям.
- Готов при необходимости работать с YAML-файлами во внешнем редакторе.

### Цели

| Цель | Описание |
|---|---|
| Схватывать идею в виде карты | Быстро переводить неоформленные идеи в визуальные структуры |
| Видеть связи между фрагментами | Понимать отношения между объектами на диаграмме |
| Выделять самостоятельные объекты | Различать устойчивые элементы в свободной диаграмме |
| Связывать уровни через детализацию | Раскрывать объекты через drill-down и собирать через step-up |
| Фиксировать повторяющиеся типы | Превращать найденные типы в reusable-нотацию |
| Формировать язык моделирования | Выращивать собственные нотации из практики моделирования |

### Боли

- Существующие инструменты требуют выбрать нотацию до начала работы.
- Свободные canvas-инструменты (Miro, FigJam) не дают семантики — нельзя извлечь нотацию, нельзя формализовать типы.
- Формальные modeler'ы слишком жёсткие для исследовательской работы.
- Нет инструмента, который поддерживает движение и вверх, и вниз по уровням абстракции.

### Ожидания от MVP

- **Простота входа**: быстро создать проект и начать рисовать без настройки.
- **Свобода первого шага**: начать с freeform-модели без обязательной нотации.
- **Поддержка уровней**: drill-down и step-up как естественные операции.
- **Сохранение структуры**: фиксация типов в виде нотации, а не только в голове.
- **Низкое сопротивление UI**: рабочая доска с семантическими возможностями, а не CASE-система.

## Контекст использования

- Личная или малокомандная песочница для проектирования и проверки идей.
- Десктоп, современные браузеры.
- Локальное использование или работа в доверенной сети.
- Параллельная работа с YAML-файлами во внешнем редакторе (VS Code) допустима, но не одновременное редактирование одного файла.

## Кто принимает результат

Заказчик принимает результат самостоятельно через ручную проверку ключевых сценариев. В качестве тестового use case может использоваться, например, Business Model Canvas с drill-down из элементов в более детальные структуры.

## Ограничения пользовательского контекста

В рамках MVP не учитываются:
- Конкурентная одновременная работа нескольких пользователей.
- Разграничение ролей и прав доступа.
- Сложные workflow согласования.
- Формальные процессы публикации нотаций.
- Автоматическая валидация корректности модели.
- Интеграция с исполняющими системами.
