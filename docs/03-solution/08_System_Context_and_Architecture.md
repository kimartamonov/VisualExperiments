# 08. System Context and Architecture

## Контекст системы

VisualExperiments — веб-приложение с клиент-серверной архитектурой, работающее локально или в доверенной сети. Внешних интеграций в MVP нет.

```
┌─────────────────┐       HTTP/REST        ┌──────────────────┐
│                 │ ◄───────────────────► │                  │
│  Browser (SPA)  │                        │  Node.js Server  │
│  React + TS     │                        │  File-based API  │
│  React Flow     │                        │                  │
│                 │                        │  ┌────────────┐  │
└─────────────────┘                        │  │ YAML Files │  │
                                           │  │ (projects, │  │
                                           │  │  models,   │  │
                                           │  │  notations)│  │
                                           │  └────────────┘  │
                                           └──────────────────┘
```

### Внешние системы

В MVP нет интеграций с внешними системами. Пользователь может параллельно редактировать YAML-файлы во внешнем редакторе (VS Code), но одновременное редактирование одного файла не является целевым сценарием.

## Технологический стек

| Компонент | Технология | Обоснование |
|-----------|-----------|-------------|
| Frontend framework | React + TypeScript | Компонентная модель, экосистема |
| Canvas | React Flow | Подтверждено прототипированием; поддерживает frame, drag-and-drop, edges |
| Backend | Node.js + TypeScript | Единый язык с frontend, достаточен для файлового API |
| Формат данных | YAML | Читаемость для человека, редактируемость в VS Code |
| Транспорт | REST API (JSON) | Простота реализации |

### Решения по стеку

- **React Flow** — предпочтительно не менять. Прототипирование подтвердило поддержку frame. Drill-down реализуется не как встроенная функция React Flow, а как ссылка в свойствах ноды.
- **TypeScript** — не является принципиальным требованием, но предпочтителен.
- **YAML** — может быть заменён на JSON или SQLite, если это улучшит архитектуру без потери читаемости.
- **База данных** — в MVP не используется. Файловое хранение достаточно.

## Архитектурные слои

### 1. Project Browser Layer
- Экран списка проектов.
- Создание и открытие проектов.
- Отображение файловой структуры проекта в левой панели.

### 2. Editor Workspace Layer
- Трёхпанельная компоновка: дерево проекта | canvas | панель свойств.
- Управление текущим контекстом (какая модель открыта, какой объект выбран).
- Навигация: хлебные крошки, кнопка «назад», дерево проекта.

### 3. Diagram Engine Layer
- React Flow как substrate для canvas.
- Отображение нод, edges, frame.
- Drag-and-drop, создание связей, контекстное меню.
- «Тонкий» canvas: React Flow отвечает за визуализацию и взаимодействие, но не за бизнес-логику.

### 4. Model Semantics Layer
- Доменная логика: создание/удаление сущностей, валидация правил.
- Операции drill-down, step-up, поздней типизации, создания нотации.
- Управление ссылками между моделями.
- Преобразование между внутренним состоянием и YAML-форматом.

### 5. Persistence Layer
- Файлово-ориентированный backend.
- CRUD-операции над проектами, моделями, нотациями через REST API.
- Чтение/запись YAML-файлов.
- Автосохранение.

---

## Взаимодействие между слоями

```
Browser (SPA)
  ├── Project Browser Layer
  │     └── → REST API → Persistence Layer
  │
  └── Editor Workspace Layer
        ├── Diagram Engine Layer (React Flow)
        │     └── events → Model Semantics Layer
        ├── Model Semantics Layer
        │     └── state changes → Diagram Engine Layer (re-render)
        │     └── save → REST API → Persistence Layer
        └── Properties Panel
              └── reads/writes → Model Semantics Layer
```

### Принцип разделения

- **Diagram Engine** (React Flow) отвечает за визуализацию и пользовательский ввод.
- **Model Semantics** отвечает за бизнес-логику, правила, преобразования.
- **Persistence** отвечает за чтение/запись файлов.
- Canvas не содержит бизнес-логики — он является substrate.

## Структура REST API

API организовано в терминах сущностей продукта.

### Проекты
- `GET /api/projects` — список проектов
- `POST /api/projects` — создать проект
- `GET /api/projects/:id` — получить manifest проекта
- `GET /api/projects/:id/tree` — файловая структура проекта

### Модели
- `GET /api/projects/:id/models/:path` — получить модель
- `POST /api/projects/:id/models` — создать модель
- `PUT /api/projects/:id/models/:path` — сохранить модель
- `DELETE /api/projects/:id/models/:path` — удалить модель *(задел, не входит в обязательный набор MVP)*

### Нотации
- `GET /api/projects/:id/notations` — список нотаций проекта
- `GET /api/projects/:id/notations/:path` — получить нотацию
- `POST /api/projects/:id/notations` — создать нотацию

### Файловые операции
- `POST /api/projects/:id/folders` — создать папку

## Управление состоянием (Frontend)

### Разделение состояния по уровням

| Уровень | Что хранит | Пример |
|---------|-----------|--------|
| App-level | Текущий проект, навигационный стек | Какой проект открыт, хлебные крошки |
| Model-level | Текущая модель, её содержимое | Ноды, edges, frames, notation ref |
| UI-level | Выделенный объект, состояние панелей | Какая нода выбрана, открыта ли панель |

Конкретный state management (Redux, Zustand, Context) — на усмотрение разработчика. Важно: архитектура должна быть расширяемой.

## Развёртывание

MVP запускается как единое приложение:
- Backend (Node.js) отдаёт статику frontend и обслуживает REST API.
- Данные хранятся в файловой системе сервера.
- Развёртывание: `npm start` локально или Docker-контейнер.

---

## Архитектурные приоритеты

1. **Простота важнее полноты** — MVP не должен содержать преждевременных абстракций.
2. **Файлы — часть архитектуры** — структура папок и YAML-файлов должна быть понятной и без UI.
3. **Один мир для freeform и typed** — единый формат хранения, единый canvas, различие только в поле `notation`.
4. **Переходы между уровнями — часть ядра** — drill-down и step-up не являются надстройками, они встроены в Model Semantics Layer.

## Технические ограничения MVP

- Нет real-time collaboration.
- Нет version control (git-интеграция).
- Нет базы данных.
- Нет runtime-компиляции в Windmill.
- Нет plugin system.

## UX-референсы

- **Sirius Web** — трёхпанельная структура, дерево проекта.
- **React Flow** — canvas, drag-and-drop.
- **FigJam / Miro** — простота визуальных элементов, низкий порог входа.
