# 09. Data and Integrations

## Файловая структура проекта

Проект хранится как папка в файловой системе сервера.

```
project-name/
  project.yaml              # manifest
  models/
    main.yaml                # модель верхнего уровня
    research/
      map-01.yaml
      map-02.yaml
    abstractions/
      upper-map.yaml
  notations/
    workflow-map.yaml
    people-needs.yaml
```

### Правила организации

- Проект = папка верхнего уровня.
- `project.yaml` — manifest, точка входа.
- `models/` — каталог моделей.
- `notations/` — каталог нотаций.
- Подпапки внутри `models/` допустимы для организации.
- Все файлы — YAML.

### Адресация

- Модели адресуются по **относительному пути** от корня проекта: `models/research/map-01.yaml`.
- Объекты адресуются по **путь модели + id**: `models/main.yaml#n1`.
- Ссылки строятся по идентификаторам, а не по именам.

## Форматы YAML-файлов

### Manifest (`project.yaml`)

```yaml
id: project-001
name: My Project
defaultModel: models/main.yaml
notations:
  - notations/workflow-map.yaml
  - notations/people-needs.yaml
```

| Поле | Обязательно | Описание |
|------|-------------|----------|
| id | Да | Уникальный идентификатор проекта |
| name | Да | Отображаемое имя |
| defaultModel | Нет | Путь к стартовой модели |
| notations | Нет | Список путей к нотациям |

### Модель

```yaml
id: model-001
name: Main Map
notation: freeform

nodes:
  - id: n1
    label: User
    description: ""
    type: actor
    position:
      x: 120
      y: 180
    drilldowns:
      - models/research/user-details.yaml

  - id: n2
    label: Need
    description: ""
    type: need
    position:
      x: 420
      y: 180
    drilldowns: []

edges:
  - id: e1
    source: n1
    target: n2
    label: ""

frames:
  - id: f1
    name: User Context
    description: ""
    nodes:
      - n1
      - n2
    stepUp:
      model: models/abstractions/context-map.yaml
      nodeId: n10
```

| Поле | Обязательно | Описание |
|------|-------------|----------|
| id | Да | Уникальный идентификатор модели |
| name | Да | Отображаемое имя |
| notation | Да | `freeform` или id нотации (не путь к файлу; соответствие id → файл определяется через список `notations` в manifest) |
| nodes | Да | Массив нод (может быть пустым) |
| edges | Да | Массив стрелок (может быть пустым) |
| frames | Да | Массив frame (может быть пустым) |

### Нотация

```yaml
id: workflow-map
name: Workflow Map
types:
  - id: actor
    name: Actor
    color: "#F5C26B"
  - id: need
    name: Need
    color: "#8ED1FC"
  - id: process
    name: Process
    color: "#C3F0CA"
recommendedDrilldowns:
  actor:
    - freeform
  process:
    - freeform
```

| Поле | Обязательно | Описание |
|------|-------------|----------|
| id | Да | Уникальный идентификатор нотации |
| name | Да | Отображаемое имя |
| types | Да | Массив типов объектов |
| recommendedDrilldowns | Нет | Задел на будущее |

## Жизненный цикл данных

### Создание

1. **Проект**: создаётся через UI → backend создаёт папку, `project.yaml`, `models/`.
2. **Модель**: создаётся через UI → backend создаёт YAML-файл в `models/`.
3. **Нотация**: создаётся через операцию «Create notation from model» → backend создаёт YAML-файл в `notations/`, обновляет manifest.

### Чтение

- Backend читает YAML-файлы с диска и передаёт данные как JSON через REST API.
- Frontend отображает данные на canvas и в панелях.

### Обновление

- Пользователь изменяет модель в UI → frontend отправляет обновлённое состояние через REST API → backend перезаписывает YAML-файл.
- Автосохранение: frontend периодически отправляет текущее состояние модели на backend.

### Удаление

- Удаление ноды: каскадное удаление связанных edges и drill-down ссылок (но не самих дочерних моделей).
- Удаление frame: ноды остаются на canvas.
- Удаление drill-down ссылки: модель не удаляется, удаляется только ссылка.

---

## Валидация данных

### При загрузке YAML

- Если YAML-файл содержит синтаксическую ошибку → система показывает понятное сообщение об ошибке и отказывает в загрузке модели.
- Если отсутствуют обязательные поля → сообщение об ошибке.
- Система не должна аварийно завершаться при некорректном YAML.

### Битые ссылки

- Если drill-down ссылается на несуществующий файл → система показывает «Модель не найдена. Создать?».
- Если step-up ссылается на несуществующую модель → аналогичное сообщение.
- Если нотация, указанная в модели, не найдена → сообщение об ошибке, модель может быть открыта как freeform.

## Обработка конфликтов

- Одновременное редактирование одного файла в VS Code и визуальном редакторе **не является целевым сценарием**.
- Автоматическое обнаружение внешних изменений в MVP не реализуется.
- При конфликте побеждает последнее сохранение (last-write-wins).

---

## Экспорт / Импорт

- В MVP нет специального механизма экспорта/импорта.
- Проект — это папка, которую можно копировать и переносить вручную.
- В перспективе возможна синхронизация с репозиторием или серверной папкой.

---

## Миграция формата

- Автоматическая миграция YAML-формата при изменении схемы в MVP **не реализуется**.
- При необходимости изменения формата файлы обновляются вручную.

---

## Интеграции

В MVP нет внешних интеграций. Система полностью автономна.

| Интеграция | Статус в MVP | Перспектива |
|-----------|-------------|-------------|
| VS Code (ручное редактирование YAML) | Неформальная, не автоматизированная | Возможна deeper интеграция |
| Git | Не реализована | Возможна синхронизация |
| Windmill | Не реализована | Генерация flows из моделей |
| Внешние API | Нет | Нет планов для MVP |

---

## Источник истины

- Источник истины для данных — **YAML-файлы на диске**.
- Frontend работает с in-memory копией, которая синхронизируется с backend через REST API.
- При перезагрузке страницы состояние восстанавливается из файлов.
